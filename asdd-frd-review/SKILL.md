# asdd-frd-review

对已生成的功能需求文档（FRD）进行审查澄清和修改调整。

## 定位

本 skill 是 asdd-frd-generate 的配套工具，在 FRD 生成之后使用。

```
asdd-frd-generate（生成 FRD）
    |
    v
docs/functional-requirements/{module}/REQ-*.md（FRD 文件）
    |
    v
asdd-frd-review（本 skill：审查 / 修改 / 一致性校验）  <-- 可多次调用
    |
    v
确认 FRD 无误后 -> 使用 asdd-detailed-design-generate skill -> 使用 asdd-tasks-generate skill -> 使用 asdd-tasks-implement skill
```

## 路径约定

| 资源 | 路径 |
|------|------|
| FRD 输出目录 | `docs/functional-requirements/{module}/` |
| FRD 索引 | `docs/functional-requirements/INDEX.md` |
| 系统需求总览 | `docs/requirements/overview.md` |
| 模块级需求 | `docs/requirements/req-modules/{module-name}.md` |
| 全局约束 | `docs/constitution.md` |
| 架构蓝图 | `docs/architecture.md` |
| 领域设计 | `docs/domains/*.md` |
| 规范约束 | `.opencode/rules/`（加载该目录下所有文件） |

## 阶段 0：前置准备

### 0.1 强制加载规范约束

扫描 `.opencode/rules/` 目录下**所有文件**，读取全部内容作为规范基准。
后续所有检测和修改都以这些规范为权威参照。

如果目录不存在或为空，向用户提示但不阻断：

```
!!! 未找到 .opencode/rules/ 目录或目录为空。
规范合规校验将跳过，仅执行文档结构和内容检测。
```

### 0.2 加载 FRD 文件

检查 `docs/functional-requirements/` 目录：

- 读取 `INDEX.md` — FRD 索引
- 扫描所有 `{module}/REQ-*.md` 文件 — FRD 内容

如果目录不存在或无 FRD 文件，中止并提示：

```
X 未找到功能需求文档（docs/functional-requirements/）。
请先使用 asdd-frd-generate skill 生成 FRD。
```

### 0.3 加载关联上下文

根据需要加载的关联文档：

1. **系统需求**：`docs/requirements/overview.md` + `req-modules/*.md`
2. **顶层设计**：`docs/constitution.md` + `docs/architecture.md` + `docs/domains/*.md`
3. **FRD 模板**：`docs/templates/functional-requirement-template.md`（如存在）

### 0.4 模式判断

根据用户意图判断工作模式：

| 用户意图 | 模式 |
|---------|------|
| 检查/审查/review/有没有问题 | **审查模式**（阶段 1 -> 2 -> 3） |
| 修改/调整/改/补充/删除某个 FRD 的具体内容 | **修改模式**（阶段 4 -> 5） |
| FRD 和系统需求对不上/一致性检查 | **系统需求一致性校验**（阶段 6 -> 2 -> 3） |
| FRD 和设计不一致/设计合规/约束冲突 | **顶层设计合规校验**（阶段 7 -> 2 -> 3） |
| FRD 不符合规范/规范校验/rules 检查 | **规范约束合规校验**（阶段 8 -> 2 -> 3） |
| 意图不明确 | 询问用户选择 |

**意图不明确时**，调用 OpenCode `question` tool 询问：

```
请选择操作模式：
  A) 审查模式 -- 全面检查 FRD 文件，发现冲突、歧义和不一致
  B) 修改模式 -- 对 FRD 进行定向修改
  C) 系统需求一致性校验 -- 将 FRD 与系统模块需求交叉比对
  D) 顶层设计合规校验 -- 将 FRD 与顶层设计文档交叉比对
```

如果用户选择 D 或提到规范相关，进一步确认是顶层设计还是规范约束：

```
请确认校验范围：
  A) 顶层设计合规 -- 与 constitution.md / architecture.md 比对
  B) 规范约束合规 -- 与 .opencode/rules/ 比对
  C) 两者都检查
```

## 审查模式

### 阶段 1：全面扫描

按以下 7 个维度扫描所有 FRD 文件，每个维度独立检测。

#### 1.1 结构完整性检测（Completeness）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| Frontmatter 必填项缺失 | requirement_id / module / source / priority / status 是否完整 | 🔴 严重 |
| 功能目标缺失 | FRD 没有功能目标描述 | 🔴 严重 |
| 业务规则缺失 | FRD 没有任何业务规则（BR-*） | 🟡 警告 |
| 验收标准缺失 | FRD 没有任何验收标准（AC-*） | 🔴 严重 |
| 影响范围缺失 | FRD 没有影响范围描述 | 🟡 警告 |
| 占位符残留 | 存在未替换的 `{{...}}` 占位符 | 🔴 严重 |
| AI 指引残留 | 存在未删除的 `<!-- AI 填充指引 -->` 注释块 | 🟡 警告 |
| 变更记录缺失 | FRD 没有变更记录表 | 🟡 警告 |

#### 1.2 编号规范检测（Numbering）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 编号格式错误 | requirement_id 不符合 `REQ-{YYYYMMDD}-{三位序号}` 格式 | 🔴 严重 |
| 编号重复 | 不同 FRD 文件使用了相同的 requirement_id | 🔴 严重 |
| 文件名与编号不一致 | 文件名中的编号与 frontmatter 中的 requirement_id 不匹配 | 🟡 警告 |
| 索引记录缺失 | FRD 文件存在但 INDEX.md 中没有对应记录 | 🟡 警告 |
| 索引幽灵记录 | INDEX.md 中有记录但对应 FRD 文件不存在 | 🔴 严重 |

#### 1.3 冲突检测（Conflicts）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 业务规则矛盾 | 不同 FRD 中的业务规则互相矛盾 | 🔴 严重 |
| 验收标准冲突 | 不同 FRD 的验收标准对同一行为有矛盾的预期 | 🔴 严重 |
| 功能重复 | 不同 FRD 描述了实质相同的功能 | 🟡 警告 |
| 优先级矛盾 | 高优先级 FRD 依赖低优先级 FRD 的功能 | 🟡 警告 |
| 模块归属冲突 | 同一功能在不同模块下有 FRD | 🔴 严重 |

#### 1.4 歧义检测（Ambiguities）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 模糊验收标准 | 验收标准缺少具体数值或明确条件（如"快速响应"而非"< 500ms"） | 🟡 警告 |
| 模糊业务规则 | 业务规则使用"适当""合理""尽量"等模糊词汇 | 🟡 警告 |
| 未定义术语 | 使用了未在系统需求或顶层设计中定义的业务术语 | 🟢 提示 |
| 边界不清 | 功能边界描述不够清晰，可能导致实现理解分歧 | 🟡 警告 |

#### 1.5 风险标记检测（Risk）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 高风险未处理 | 风险标记中存在高风险（RD1-RD6）但没有应对方案 | 🟡 警告 |
| 跨模块影响未声明 | FRD 涉及多模块但未标记 RD4 风险 | 🟡 警告 |
| 性能风险未量化 | RD6 性能风险存在但缺少具体指标 | 🟢 提示 |

#### 1.6 跨 FRD 一致性检测（Cross-FRD）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 依赖 FRD 不存在 | 影响范围中引用的 FRD 编号不存在 | 🔴 严重 |
| 约束引用无效 | related_constraints 中引用的约束 ID 在 constitution.md 中不存在 | 🟡 警告 |
| 模块目录不一致 | FRD 的 module 字段与其所在目录名不一致 | 🟡 警告 |

#### 1.7 内容质量检测（Quality）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 包含实现细节 | FRD 中包含代码片段、SQL、API 定义等实现细节 | 🟡 警告 |
| 功能目标过于笼统 | 功能目标描述不够具体，无法指导后续设计 | 🟡 警告 |
| 验收标准不可测试 | 验收标准描述无法转化为测试用例 | 🟡 警告 |

### 阶段 2：问题报告与用户澄清

#### 2.1 问题汇总

将检测到的问题按严重度分批展示。

**第一批：严重问题（🔴）** -- 必须逐项确认：

```
🔴 发现 {N} 个严重问题，需要逐项确认：

--------------------------------------------
[C1] 业务规则矛盾
--------------------------------------------
FRD REQ-20260401-001（用户登录）的 BR-003 要求"密码错误 5 次锁定账户 30 分钟"，
但 REQ-20260401-005（安全策略）的 BR-002 要求"密码错误 3 次锁定账户 1 小时"。

位置：
  - docs/functional-requirements/user-auth/REQ-20260401-001-user-login.md BR-003
  - docs/functional-requirements/user-auth/REQ-20260401-005-security-policy.md BR-002
```

对每个严重问题，调用 OpenCode `question` tool 提供修复选项：

```
如何处理密码锁定策略冲突？
  A) 统一为 5 次/30 分钟（以 REQ-001 为准）
  B) 统一为 3 次/1 小时（以 REQ-005 为准）
  C) 自定义策略
  D) 跳过 -- 暂不处理
```

**第二批：警告问题（🟡）** -- 支持批量确认：

```
🟡 发现 {N} 个警告问题：

[W1] 模糊验收标准
  - REQ-20260401-003 AC-002: "系统应快速响应" -> 建议改为 "响应时间 < 500ms"
  位置：docs/functional-requirements/order-flow/REQ-20260401-003-order-create.md

[W2] FRD 包含实现细节
  - REQ-20260401-002 包含 SQL 片段，建议移除
  位置：docs/functional-requirements/user-auth/REQ-20260401-002-token-mgmt.md

...

请逐项确认，或输入"全部接受建议"批量处理。
```

**第三批：提示信息（🟢）** -- 仅展示，不要求确认：

```
🟢 {N} 条提示信息（仅供参考，无需确认）：
  [I1] 术语"风控评分"仅在 REQ-20260401-010 中出现，建议在系统需求中定义
  [I2] REQ-20260401-007 的 RD6 性能风险缺少具体并发指标
```

#### 2.2 无问题时的处理

```
OK FRD 审查完成，未发现问题。

扫描范围：
  - docs/functional-requirements/ 下 {N} 个模块、{M} 个 FRD 文件
  - INDEX.md 索引

检测维度：完整性 / 编号规范 / 冲突 / 歧义 / 风险标记 / 跨 FRD 一致性 / 内容质量
结果：全部通过

FRD 可以进入下一步流程（asdd-detailed-design-generate）。
```

### 阶段 3：自动修复

#### 3.1 收集用户决策

将用户的所有决策汇总展示：

```
修复计划：

| 编号 | 问题 | 用户决策 | 影响文件 |
|------|------|---------|---------|
| C1 | 密码锁定策略冲突 | 统一为 3 次/1 小时 | REQ-001, REQ-005 |
| W1 | 模糊验收标准 | 改为 < 500ms | REQ-003 |
| W2 | 包含实现细节 | 移除 SQL 片段 | REQ-002 |

确认执行以上修复？
```

#### 3.2 执行修复

用户确认后，按以下顺序修复：

1. **FRD 文件** -- 修改各 FRD 中的具体内容
2. **INDEX.md** -- 同步更新索引信息（如有变更）
3. **变更记录** -- 在每个被修改的 FRD 的变更记录表追加一行

#### 3.3 修复验证

修复完成后，对修改的文件重新执行相关检测项，确认问题已解决。

```
OK 审查修复完成

已修复：{N} 项问题
跳过：{M} 项（用户选择暂不处理）

已修改文件：
  - docs/functional-requirements/user-auth/REQ-20260401-001-user-login.md
  - docs/functional-requirements/user-auth/REQ-20260401-005-security-policy.md
  - docs/functional-requirements/order-flow/REQ-20260401-003-order-create.md

验证结果：修复后的文件通过所有相关检测
```

## 修改模式

### 阶段 4：理解修改意图

#### 4.1 定位修改目标

根据用户输入定位要修改的 FRD：

| 用户说 | 定位 |
|--------|------|
| "修改 REQ-20260401-001" | -> 通过编号定位具体 FRD 文件 |
| "修改用户登录的 FRD" | -> 通过功能名搜索匹配 FRD |
| "修改 user-auth 模块的 FRD" | -> 列出该模块下所有 FRD 供选择 |
| "修改某个 FRD 的业务规则" | -> 定位到该 FRD 的业务规则 section |
| "修改某个 FRD 的验收标准" | -> 定位到该 FRD 的验收标准 section |

**如果用户未明确指定目标**，列出所有 FRD 供选择：

```
请选择要修改的 FRD：

| 模块 | 编号 | 功能名称 | 优先级 | 状态 |
|------|------|---------|--------|------|
| user-auth | REQ-20260401-001 | 用户登录 | P0 | 🔴 待处理 |
| user-auth | REQ-20260401-002 | Token 管理 | P1 | 🔴 待处理 |
| order-flow | REQ-20260401-003 | 创建订单 | P0 | 🔴 待处理 |

请输入编号或序号选择：
```

#### 4.2 理解修改内容

用户的修改请求分类及影响范围：

| 修改类型 | FRD 影响 | 可能的上游影响 |
|---------|---------|--------------|
| **修改业务规则** | 目标 FRD 的 BR-* | 如果规则变更影响系统需求中的功能描述 -> 提示同步 |
| **修改验收标准** | 目标 FRD 的 AC-* | 如果标准变更影响系统需求中的验收标准 -> 提示同步 |
| **修改功能目标** | 目标 FRD 功能目标 section | 如果目标变更影响系统需求中的专题目标 -> 提示同步 |
| **修改优先级** | 目标 FRD frontmatter | 如果与系统需求中的优先级不一致 -> 提示同步 |
| **修改风险标记** | 目标 FRD 风险标记 section | 无 |
| **修改影响范围** | 目标 FRD 影响范围 section | 无 |
| **新增业务规则/验收标准** | 目标 FRD 追加内容 | 可能需要更新系统需求的功能描述 |
| **删除业务规则/验收标准** | 目标 FRD 移除内容 | 如果原内容来源于系统需求 -> 提示用户确认 |

#### 4.3 上游影响评估

**这是本模式的核心检查点。** 每次修改 FRD 时，都要评估是否影响上游文档：

1. **系统模块需求影响**：将修改内容与 `docs/requirements/req-modules/{module}.md` 中对应专题/功能点比对：
   - 如果 FRD 的业务规则变更导致系统需求的功能描述不再准确 -> 提示
   - 如果 FRD 的验收标准变更导致系统需求的验收标准过时 -> 提示
   - 如果 FRD 的功能目标变更导致系统需求的专题目标不一致 -> 提示

2. **顶层设计影响**：将修改内容与 `constitution.md` / `architecture.md` 比对：
   - 如果修改涉及新的技术约束（如新增对某中间件的依赖） -> 提示
   - 如果修改改变了模块间的交互方式 -> 提示

**提示格式**：

```
!! 上游影响提醒

本次 FRD 修改可能影响以下上游文档：

1. 系统模块需求：
   - docs/requirements/req-modules/user-auth.md T1 F1.2 功能描述
   - 原因：FRD BR-003 的锁定次数从 5 次改为 3 次，但系统需求中仍描述为 5 次

2. 顶层设计：
   - docs/constitution.md E2 安全约束
   - 原因：新增的 MFA 要求可能需要更新安全约束

建议操作：
  A) 继续修改 FRD，稍后手动同步上游文档
  B) 继续修改 FRD，并自动同步上游文档中的对应内容
  C) 暂停修改，先确认上游文档的调整方案
```

### 阶段 5：执行修改

#### 5.1 展示修改计划

```
修改计划：

**目标 FRD**：REQ-20260401-001（用户登录）
**修改类型**：业务规则修改

**具体变更**：
  1. docs/functional-requirements/user-auth/REQ-20260401-001-user-login.md:
     - BR-003: "密码错误 5 次锁定 30 分钟" -> "密码错误 3 次锁定 1 小时"
     - 新增 BR-006: "锁定期间显示剩余解锁时间"
     - AC-005: 同步更新验收标准

  2. 上游同步（用户选择自动同步时）：
     - docs/requirements/req-modules/user-auth.md T1 F1.2: 更新锁定策略描述

确认执行？
```

#### 5.2 执行修改

用户确认后：

1. 修改目标 FRD 文件内容
2. 同步更新 INDEX.md（如有编号/状态/名称变更）
3. 如用户选择自动同步，更新上游文档
4. 在 FRD 的变更记录表追加一行
5. 如同步了上游文档，在上游文档的变更记录/历史表中也追加记录

#### 5.3 修改验证

修改完成后，对受影响的文件执行相关检测，确保修改未引入新问题：

```
OK 修改完成
```

## 系统需求一致性校验

### 阶段 6：交叉比对

从以下 5 个维度将 FRD 与系统模块需求进行比对。

#### 6.1 覆盖完整性检测

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 专题未覆盖 | 系统需求中的专题没有对应的 FRD | 🟡 警告 |
| 功能点未覆盖 | 系统需求中的功能点没有在任何 FRD 中体现 | 🟡 警告 |
| FRD 无源 | FRD 的 source_topic 在系统需求中找不到对应专题 | 🟡 警告 |
| 模块不匹配 | FRD 的 module 与系统需求中该专题所属模块不一致 | 🔴 严重 |

#### 6.2 内容一致性检测

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 功能目标偏差 | FRD 功能目标与系统需求专题目标明显不一致 | 🔴 严重 |
| 验收标准矛盾 | FRD 验收标准与系统需求验收标准数值或条件矛盾 | 🔴 严重 |
| 功能范围超出 | FRD 包含了系统需求中未描述的功能 | 🟡 警告 |
| 功能范围缩减 | FRD 遗漏了系统需求中已描述的功能点 | 🟡 警告 |

#### 6.3 优先级一致性检测

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 优先级不一致 | FRD 优先级与系统需求中对应专题/模块的优先级不一致 | 🟡 警告 |
| 依赖优先级倒挂 | 高优先级 FRD 依赖的功能在低优先级系统需求专题中 | 🟡 警告 |

#### 6.4 实体一致性检测

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 实体名称不一致 | FRD 中使用的实体名称与系统需求中的核心实体名称不同 | 🟡 警告 |
| 新增实体未声明 | FRD 引入了系统需求中未声明的新实体 | 🟢 提示 |

#### 6.5 依赖一致性检测

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 跨模块依赖不一致 | FRD 影响范围中的模块依赖与系统需求依赖图不一致 | 🟡 警告 |
| 新增依赖未声明 | FRD 引入了系统需求中未声明的模块间依赖 | 🟡 警告 |

### 阶段 7：设计合规比对

## 关键原则

### 修改必须评估上游影响

每次修改 FRD 时，必须检查是否影响系统模块需求和顶层设计。这是本 skill 与 asdd-requirements-review 的关键区别 -- FRD 是下游文档，修改时要向上看。

### 规范约束是权威参照

`.opencode/rules/` 中的规范具有最高权威性。当 FRD 与规范冲突时，默认建议以规范为准。但最终决策权在用户 -- 如果用户明确要求偏离规范，应提醒风险但执行修改。

### 非破坏性修改

- 修改业务规则/验收标准时，编号不重排（BR-001, BR-002, BR-004 -- 保留空位），确保外部引用稳定
- 删除 FRD 时，不影响其他 FRD 的编号
- 已进入设计阶段（docs/modules/ 已生成）的 FRD，修改时需提示用户同步更新设计文档

### 修改追溯

所有修改都记录在被修改文件的变更记录表中：

```markdown
| 日期 | 变更内容 | 作者 |
|------|----------|------|

| 2026-04-03 | 手动修改：新增 MFA 业务规则 | AI |
```
