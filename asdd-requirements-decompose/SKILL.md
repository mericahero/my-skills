# asdd-requirements-decompose

将系统级大需求拆解为微服务级（模块级）需求，支持多轮增量输入。

## 定位

本 skill 解决的是**新建系统**场景下的第一步问题：拿到一份大需求后，如何划分模块、规划微服务、拆分需求并分发给各开发团队。

拆解完成后，各团队认领模块，按以下 skill 顺序继续推进：使用 asdd-architecture-generate skill → 使用 asdd-frd-generate skill → 使用 asdd-detailed-design-generate skill → 使用 asdd-tasks-generate skill → 使用 asdd-tasks-implement skill。

```
                          本 skill 的位置
                          ─────────────────
用户多轮输入系统级需求（对话中）
    │
    ▼
asdd-requirements-decompose（本 skill，多次调用）
    │
    ▼
<项目根目录>/docs/requirements/
├── overview.md
└── req-modules/*.md          ← 各团队认领
    │
    │  🧑 人工评审 + 团队认领
    ▼
使用 asdd-architecture-generate skill          ← 已有 skill
    │
    ▼
使用 asdd-frd-generate skill（per topic）      ← 已有 skill
    │
    ▼
使用 asdd-detailed-design-generate skill（per module）   ← 已有 skill
    │
    ▼
使用 asdd-tasks-generate skill → 使用 asdd-tasks-implement skill ← 已有流程
```

## 路径约定

- **输出目录**：`<项目根目录>/docs/requirements/`
- **总览文件**：`<项目根目录>/docs/requirements/overview.md`
- **模块需求目录**：`<项目根目录>/docs/requirements/req-modules/`

## 阶段 1：需求分析

对用户输入的系统级需求进行深度分析。

### 1.1 需求理解

从用户输入中提取以下信息（用户输入无固定格式，AI 自行理解和提炼）：

- **系统定位**：系统解决什么问题、服务谁、核心价值
- **功能清单**：所有功能点（粗粒度列表）
- **数据实体**：涉及的核心数据对象及其关系
- **业务流程**：核心业务链路（如：下单→支付→发货→收货）
- **非功能需求**：性能、安全、可用性等要求
- **外部依赖**：依赖的外部系统或第三方服务
- **系统边界**：明确做什么、不做什么

### 1.2 系统定位推断与完整能力域识别

**这是确保模块拆分合理性的关键步骤。**

在理解用户输入的需求内容后，必须执行以下推断流程：

#### 1.2.1 推断系统类型

根据用户输入的需求内容，推断系统的类型和定位（如：企业管理系统、电商平台、SaaS 产品、IoT 平台、内容平台等）。

#### 1.2.2 推断完整能力域

基于推断出的系统类型，结合软件工程最佳实践，列出**该类系统通常具备的完整能力域**。能力域分为两类：

- **通用支撑能力域**：几乎所有系统都需要的基础能力，如用户管理、鉴权与授权、角色权限、审计日志、通知服务等
- **业务能力域**：与系统类型强相关的业务能力，如电商的商品管理、订单流程、支付结算等

列出时需明确标注哪些是当前输入已覆盖的，哪些是推断补充的：

```
🟢 已覆盖（当前输入已包含具体需求）：
  - {能力域 1}
  - {能力域 2}

🟡 推断补充（该类系统通常需要，当前输入未涉及）：
  - {能力域 A}（理由：{为什么该类系统通常需要}）
  - {能力域 B}（理由：...）
```

#### 1.2.3 与用户确认能力域

**必须等待用户确认后再进入拆解。** 用户可能的操作：
- 确认全部纳入
- 移除部分推断补充的能力域（"这个系统不需要通知服务"）
- 补充 AI 未推断到的能力域
- 调整能力域的定位或范围

确认后的完整能力域列表将作为阶段 2 模块拆解的输入。

### 1.3 歧义与缺失标记

分析过程中，标记以下问题：

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| 歧义 | 需求描述模糊，可有多种理解 | 列出可能的理解，请用户澄清 |
| 缺失 | 明显缺少的需求（如提到了审批但没有权限管理） | 提示用户是否需要补充 |
| 矛盾 | 需求内部矛盾（如两处对同一功能的描述不一致） | 列出矛盾点，请用户裁决 |
| 超界 | 功能点超出系统定位范围 | 提示用户确认是否需要 |

**增量模式额外检查**：
- 新需求与已有模块的重叠或冲突
- 新需求是否属于已有模块的延伸（包括由系统定位推断建立的模块），还是需要新建模块
- 新需求是否改变已有模块的边界或依赖关系
- 新需求是否为已有模块补充了更详细的专题和功能点（将概要内容升级为完整内容）

**增量模式下不再重复执行 1.2 系统定位推断**，因为完整能力域已在初始化时确定。但如果新需求揭示了新的能力域（初始化时未被推断到），应建议新增模块。

### 1.4 与用户确认分析结果

将分析结果展示给用户，等待确认后再进入拆解阶段。

重点展示：
- 提炼的系统定位和功能清单（用户确认是否理解正确）
- 系统定位推断出的完整能力域（用户确认纳入范围）
- 标记的歧义/缺失/矛盾项（用户逐一澄清）

```
📋 需求分析结果：

**系统定位**：{AI 提炼的系统定位}
**系统类型**：{推断的系统类型，如"企业管理系统"}

**识别到 {N} 个功能域**（来自当前输入）：
1. {功能域 1} — {核心功能点列表}
2. {功能域 2} — {核心功能点列表}
...

**推断补充的能力域**（{M} 个，基于该类系统通常具备的能力）：
1. 🟡 {能力域 A} — {该类系统通常需要此能力的理由}
2. 🟡 {能力域 B} — {理由}
...

**核心数据实体**：{实体列表及关系}

**待澄清项**（{K} 项）：
⚠️ Q1: {歧义/缺失/矛盾描述}
⚠️ Q2: {歧义/缺失/矛盾描述}

请确认：
1. 系统定位和类型是否正确？
2. 推断补充的能力域中，哪些需要纳入？哪些不需要？
3. 回答待澄清项。
```

## 阶段 2：模块拆解

基于确认后的需求分析和完整能力域列表，进行模块划分。

### 2.1 拆解策略

模块拆解分为两步：

#### Step 1 — 基于完整能力域建立模块结构

以阶段 1.2 确认后的完整能力域列表（包括用户输入已覆盖的 + 推断补充并经用户确认的）为基础，划分模块。这确保了即使当前输入只覆盖了部分需求，拆出的模块结构也是符合软件工程逻辑的完整结构。

#### Step 2 — 将当前输入的需求内容映射到对应模块

将用户当前输入的具体需求内容（功能点、业务规则、验收标准等），分配到 Step 1 建立的对应模块中，生成专题和功能点。

- **当前输入已覆盖的模块**：正常生成完整的专题（T1、T2...）、功能点（F1.1、F1.2...）和验收标准
- **推断补充的模块（当前输入未涉及具体需求）**：基于能力域的定义和系统定位，生成合理的专题方向和预期功能点概要，标注需求来源为"系统定位推断"，后续增量输入时再补充详细内容

#### Java 微服务拆分原则（前置约束）

在进行模块边界判断之前，必须先通过以下微服务拆分原则校验。这些原则来源于 Java/Spring Cloud 微服务架构的最佳实践和 DDD（领域驱动设计）方法论，是模块拆分的硬性约束：

| 原则 | 要求 | 违反时的处理 |
|------|------|------------|
| **DDD 限界上下文对齐** | 每个微服务对应一个限界上下文（Bounded Context），模块边界必须与领域边界一致，不得按技术层（Controller/Service/DAO）拆分 | 重新识别领域边界，按聚合根归属划分模块 |
| **单一职责原则（SRP for Services）** | 一个服务只负责一个业务能力域，变更原因应当单一。不允许出现"大杂烩"服务 | 拆分为多个职责清晰的服务 |
| **数据库独占原则** | 每个服务独占数据存储，禁止跨服务直连数据库（与 constitution.md S2 对齐） | 通过 API 或事件机制解耦数据访问 |
| **服务粒度控制** | 过细检测：服务仅 1-2 个 Topic 且无独立数据实体 → 建议合并；过粗检测：服务超过 8 个 Topic 或跨越 3 个以上子域 → 建议拆分；参考指标：一个服务 3-5 人团队可完整负责 | 合并过细服务或拆分过粗服务 |
| **通信模式约束** | 同步调用（Feign）仅用于需立即返回结果的场景；跨服务状态变更优先使用异步事件（RocketMQ）；同步调用超时 ≤ 3s（与 constitution.md M6 对齐） | 调整通信方式，长耗时操作改为异步 |
| **避免分布式单体** | 禁止以下反模式：①服务间同步调用链过长（A→B→C→D，超过 3 层）②跨服务共享数据模型/实体 ③多个服务必须同步部署上线 | 引入事件驱动解耦，拆除共享模型 |
| **基础设施服务分离** | 通用能力（认证鉴权、API 网关、消息中间件、日志审计、配置中心）必须独立为基础设施服务，不与业务服务混合 | 将基础设施能力抽离为独立服务 |

#### 模块边界判断维度

在满足微服务拆分原则的前提下，按以下维度综合判断模块边界：

| 维度 | 判断规则 | 权重 |
|------|---------|------|
| **数据归属** | 核心实体归属哪个模块？一个实体只能归属一个模块 | 高 |
| **业务内聚** | 功能点是否围绕同一业务流程？高内聚的功能归为同一模块 | 高 |
| **变更频率** | 变更频率相近的功能归为同一模块，避免高频变更影响稳定模块 | 中 |
| **团队边界** | 模块粒度应适合一个小团队（3-5 人）独立开发 | 中 |
| **部署独立性** | 模块应能独立部署、独立扩缩容 | 中 |
| **依赖方向** | 避免循环依赖，依赖方向应单向或分层 | 高 |

### 2.2 模块划分

对每个识别出的模块，确定：

- **模块名**：英文 kebab-case（如 `user-auth`、`order-flow`）
- **模块职责**：一句话描述
- **模块类型**：全栈 / 纯后端 / 纯前端
- **优先级**：P0（阻塞上线）/ P1（核心功能）/ P2（增强功能）
- **核心实体**：模块管辖的数据对象
- **建议微服务**：该模块建议归属或独立为哪个微服务
- **限界上下文**：该模块对应的 DDD 限界上下文名称（确保模块与限界上下文一一对应）
- **通信模式**：与其他服务的主要通信模式（sync-Feign / async-MQ / event-driven），每对依赖关系都需标注
- **反模式检查**：是否命中分布式单体反模式（✅ 通过 / ⚠️ 告警 + 原因）
- **功能点映射**：原始需求中哪些功能点归属此模块

### 2.3 微服务拓扑规划

基于模块划分，建议微服务架构：

- 哪些模块合并为同一个微服务（模块粒度可能比服务粒度更细）
- 服务间通信方式（同步 REST/Feign vs 异步 MQ）
- 网关、注册中心等基础设施需求
- 用 ASCII Art 绘制服务拓扑图

### 2.4 模块依赖分析

分析模块间的依赖关系：

- 调用方向（谁调用谁）
- 通信方式（同步/异步）
- 依赖强度（强依赖 = 必须可用 / 弱依赖 = 可降级）
- 用 ASCII Art 绘制依赖关系图

### 2.5 风险识别

| 风险类型 | 说明 |
|---------|------|
| 模块边界模糊 | 某些功能可归属多个模块 |
| 循环依赖 | 模块间存在循环调用 |
| 过度拆分 | 模块过多，增加协调成本 |
| 拆分不足 | 单个模块过于庞大 |
| 数据归属争议 | 某些实体被多个模块需要 |
| 优先级依赖 | P1 模块依赖 P2 模块 |

### 2.6 与用户确认拆解方案

将完整的拆解方案展示给用户，这是最关键的确认点。

```
📋 模块拆解方案：

**模块划分**（共 {N} 个模块）：

| 模块 | 职责 | 类型 | 优先级 | 核心实体 | 限界上下文 | 通信模式 | 建议微服务 | 反模式检查 |
|------|------|------|--------|---------|-----------|---------|-----------|-----------|
| user-auth | 用户认证与授权 | 全栈 | P0 | User, Role, Token | 用户身份域 | sync-Feign | user-service | ✅ 通过 |
| order-flow | 订单全生命周期 | 全栈 | P0 | Order, Payment | 订单域 | async-MQ | order-service | ✅ 通过 |
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

**微服务拓扑**：
{ASCII 拓扑图}

**模块依赖关系**：
{ASCII 依赖图}

**风险项**（{M} 项）：
⚠️ R1: {风险描述}
⚠️ R2: {风险描述}

请确认模块划分方案，或提出调整意见。
```

**必须等待用户确认后才能进入文件生成阶段。**

## 阶段 3：生成需求文件

基于确认后的拆解方案，生成产物文件。

### 3.1 创建目录结构

检查模板是否存在：

- `docs/templates/requirements-overview-template.md`
- `docs/templates/requirements-module-template.md`

缺失则中止并提示补充模板。创建目录结构：

```
<项目根目录>/docs/requirements/
├── overview.md
└── req-modules/
    ├── {module-1}.md
    ├── {module-2}.md
    └── ...
```

### 3.2 生成 overview.md

从 `docs/templates/requirements-overview-template.md` 复制生成 `docs/requirements/overview.md`。overview.md 是拆解产物的总览文件，必须覆盖以下内容：

**必须覆盖项**：

1. **系统定位与目标** — 系统解决什么问题、服务谁、核心价值、预期规模
2. **模块划分总表** — 包含以下列：

   | 模块 | 职责 | 类型 | 优先级 | 核心实体 | 限界上下文 | 专题数 | 通信模式 | 建议微服务 | 反模式检查 | 认领状态 |
   |------|------|------|--------|---------|-----------|--------|---------|-----------|-----------|---------|
   
   认领状态：🔴 待认领 / 🟡 已认领 / 🟢 设计中 / 🔵 已完成

3. **微服务拓扑建议** — ASCII Art 服务拓扑图 + 服务清单表
4. **模块依赖关系** — ASCII Art 依赖图 + 依赖明细表
5. **系统边界** — 包含与不包含
6. **非功能需求** — 性能、安全、可用性等
7. **外部依赖** — 依赖的外部系统
8. **风险与歧义清单** — 跨模块的风险、待确认事项
9. **拆解历史** — 记录每轮输入和产生的变更：

   | 轮次 | 日期 | 输入摘要 | 变更内容 |
   |------|------|---------|---------|

### 3.3 生成 req-modules/*.md

每个模块一个文件，文件名为模块英文名（如 `user-auth.md`）。

从 `docs/templates/requirements-module-template.md` 复制生成 `docs/requirements/req-modules/{module}.md`，再按模块实际需求扩展内容。一个简单模块可能 50 行，一个复杂模块可能 300 行，但必须保留模板来源版本 frontmatter，并覆盖以下内容：

**最低覆盖检查清单**：

- [ ] **模块目标** — 这个模块解决什么问题、核心价值（1-2 段话）
- [ ] **专题索引表** — 模块内所有专题的索引（见下方结构说明）
- [ ] **功能专题** — 按业务专题组织功能点，每个专题可独立生成 FRD（见下方结构说明）
- [ ] **核心实体** — 模块管辖的数据对象列表及关键属性
- [ ] **模块依赖** — 上游依赖（我调用谁）和下游依赖（谁调用我）
- [ ] **优先级** — P0/P1/P2 及理由
- [ ] **需求来源追溯** — 来自第几轮输入的哪部分内容（支持增量追溯）

#### 功能专题结构（核心设计）

模块内的功能清单**必须按专题（Topic）组织**，每个专题是一个可独立交付的功能板块。这是支持下游迭代开发的关键结构——团队认领模块后，可以按专题逐个生成 FRD，逐个走 asdd-detailed-design-generate 流程，而不必一次性处理整个模块。

**专题索引表**：每个模块文件必须包含一个专题索引表，位于文件顶部（模块目标之后）：

```markdown
## 专题索引

| 专题 | 名称 | 优先级 | 核心功能数 | 状态 |
|------|------|--------|-----------|------|
| T1 | 用户登录 | P0 | 4 | 🔴 待处理 |
| T2 | Token 管理 | P0 | 3 | 🔴 待处理 |
| T3 | 权限管理 | P1 | 5 | 🔴 待处理 |
| T4 | 用户信息管理 | P1 | 3 | 🔴 待处理 |
```

专题生命周期状态：🔴 待处理 / 🟡 进行中 / 🟢 待关闭 / 🔵 已关闭
FRD 生成中、设计中、实现中等具体阶段由下游 FRD / specs / tasks 目录推断。

**每个专题的内容结构**：

```markdown
### T1 用户登录

**目标**：{专题解决什么问题}

**功能点**：
- F1.1 用户名密码登录 — {描述} — {关键业务规则}
- F1.2 验证码校验 — {描述} — {关键业务规则}
- F1.3 登录失败锁定 — {描述} — {关键业务规则}
- F1.4 记住登录状态 — {描述} — {关键业务规则}

**涉及实体**：User, LoginLog

**验收标准**：
- AC-001: WHEN 用户输入正确用户名密码 THEN 登录成功，返回 Token
- AC-002: WHEN 连续 5 次登录失败 THEN 账号锁定 30 分钟
- ...

**需求来源**：第 1 轮输入 §3.2 用户认证部分
```

**专题编号规则**：
- 格式：`T{N}`（如 T1、T2、T3），模块内唯一
- 功能点编号：`F{专题号}.{序号}`（如 F1.1、F1.2），专题内唯一
- 增量模式下新增的专题从已有最大编号 +1 开始

**下游对接方式**：

用户可以按专题粒度驱动后续流程：
```
"把 user-auth 的 T1 用户登录生成 FRD"
  → asdd-frd-generate 读取 req-modules/user-auth.md 的 T1 专题内容作为输入
  → 生成 FRD：REQ-YYYYMMDD-NNN-user-login.md

"把 user-auth 的 T2 Token 管理生成 FRD"
  → asdd-frd-generate 读取 req-modules/user-auth.md 的 T2 专题内容作为输入
  → 生成 FRD：REQ-YYYYMMDD-NNN-token-mgmt.md
```

也可以将多个专题合并生成一个 FRD（如 T1 + T2 合为一个认证相关的 FRD），由用户决定粒度。

**专题划分原则**：
- 每个专题围绕一个**独立可交付的业务场景**，不是技术层级
- 专题间可以有依赖（如 T2 依赖 T1），但各自可独立定义验收标准
- 专题粒度目标：一个专题生成的 FRD 对应 1-2 周的开发量
- 过大的专题（功能点 > 10）应拆分为多个专题
- 过小的专题（功能点 < 2）应合并到相关专题

**内容自由度说明**：

以上是最低结构要求。AI 可以根据专题的复杂度，在专题内自由增加以下内容（不限于此列表）：
- 核心业务流程图（ASCII Art）
- 状态机描述
- 权限矩阵
- 数据量预估
- 性能要求
- 异常场景
- 集成要求

关键原则：内容服务于帮助认领团队理解"要做什么"，而不是"怎么做"。"怎么做"是后续 asdd-detailed-design-generate 的职责。

### 3.4 增量模式的文件处理

增量模式下，对已有文件的处理规则：

| 场景 | 处理方式 |
|------|---------|
| 新需求产生新模块 | 创建新的 `req-modules/{module}.md` |
| 新需求扩展已有模块 | 更新已有文件：在已有专题中追加功能点，或新增专题（编号从最大 +1），更新专题索引表和需求来源 |
| 新需求为推断建立的模块补充详细需求 | 更新已有文件：将概要性的专题方向升级为完整的专题内容（功能点、验收标准等），更新需求来源 |
| 新需求改变模块边界 | 更新受影响的模块文件 + overview.md 的模块划分表 |
| 新需求新增依赖关系 | 更新 overview.md 的依赖图 + 相关模块的依赖声明 |

overview.md 的拆解历史表必须追加新行，记录本轮变更。

### 3.5 重构模式的文件处理

| 场景 | 处理方式 |
|------|---------|
| 合并模块 | 创建合并后的文件，删除原文件，更新 overview.md |
| 拆分模块 | 创建拆分后的多个文件，删除原文件，更新 overview.md |
| 重命名模块 | 重命名文件，更新 overview.md 和其他模块的依赖引用 |
| 调整边界 | 在相关模块间迁移功能点，更新依赖关系 |
| 删除模块 | 删除文件，将功能点标记为"已移除"或迁移到其他模块 |

### 4.2 质量自检

文件生成后，自动执行以下检查：

| 检查项 | 规则 |
|--------|------|
| 功能覆盖 | 原始需求中的所有功能点都被分配到至少一个模块的某个专题中 |
| 实体归属 | 每个核心实体有且只有一个主管模块 |
| 依赖一致 | overview.md 的依赖图与各模块文件中的依赖声明一致 |
| 无遗漏模块 | overview.md 模块表中的模块与 req-modules/ 目录下的文件一一对应 |
| 专题索引一致 | 每个模块文件的专题索引表与文件内的专题内容一一对应 |
| 专题编号唯一 | 每个模块内的专题编号（T1、T2...）和功能点编号（F1.1、F1.2...）无重复 |
| 最低覆盖 | 每个模块文件满足最低覆盖检查清单的所有项 |
| 需求追溯 | 每个专题的功能点都能追溯到需求来源 |
| 限界上下文完整性 | 每个模块都标注了限界上下文，且无两个模块共享同一限界上下文 |
| 服务粒度合理性 | 无过细（< 2 Topic 且无独立实体）或过粗（> 8 Topic 或跨越 3+ 子域）的模块 |
| 同步调用链长度 | 任意同步调用链（Feign）不超过 3 层 |
| 分布式单体检测 | 无共享实体归属、无跨服务共享数据模型、无强制同步部署依赖 |
| 通信模式声明 | 每对模块间依赖关系都标注了通信模式（sync-Feign / async-MQ / event-driven） |
| 模板来源元数据 | overview.md 和 req-modules/*.md 保留 `template_id`、`template_version`、`target_path` |

检查不通过的项自动修复，修复后向用户报告。

## 关键原则

### 需求层 vs 设计层边界

本 skill 的产出是**需求层**文件，定义"做什么"和"为什么做"。严格不包含以下设计层内容：

| 不包含 | 归属 |
|--------|------|
| API 接口定义（路径、参数、响应） | asdd-detailed-design-generate → specs/{REQ}/api-design.md |
| 数据库表结构（DDL、字段定义） | asdd-detailed-design-generate → specs/{REQ}/backend-database-design.md |
| 代码架构（类图、包结构） | asdd-detailed-design-generate → specs/{REQ}/backend-detailed-design.md |
| 页面布局（组件、交互细节） | asdd-detailed-design-generate → specs/{REQ}/frontend-page-design.md |
| 技术选型细节 | asdd-architecture-generate → constitution.md |

### 模块命名规范

- 使用英文 kebab-case：`user-auth`、`order-flow`、`risk-engine`
- 名称应体现业务含义，不体现技术实现
- 与后续 `<项目根目录>/docs/modules/{module}/` 的模块名保持一致

### 增量一致性

多轮输入后，产出文件必须保持整体一致性：
- 模块间依赖无循环
- 功能点不重复分配
- 实体归属不冲突
- overview.md 与 req-modules/*.md 数据同步
