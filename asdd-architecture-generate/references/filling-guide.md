# 逐模板填充指南

本文档定义了每个模板文件的具体填充规则、格式要求和示例。填充顺序严格按 constitution.md → architecture.md → unlisted-components.md → 领域文件 → INDEX.md 执行。

---

## 通用规则

### 占位符替换

所有 `{{...}}` 占位符必须替换为实际内容。常见占位符：

| 占位符 | 替换为 |
|--------|--------|
| `{{项目名称}}` | 从构建文件（artifactId/name）、README 标题或需求文件提取 |
| `{{YYYY-MM-DD}}` | 当前日期，格式如 `2025-03-15`（使用执行当天的实际日期） |
| `{{🟡 设计中}}` | 保持 `🟡 设计中`（自动生成的文档默认此状态） |
| `{{🔴 待设计}}` | 更新为 `🟡 设计中` |

### AI 指引注释块删除

删除所有以下模式的注释块：

```
<!--
AI 填充指引：
...（任意内容）
-->
```

以及其他指导性注释块，识别特征：
- 以 `<!--` 开头
- 包含"AI 填充指引"、"创建前先判断"、"仅当决策涉及"、"本章节只记录"、"给关键设计阶段的指引"、"按需扩展分类"、"领域设计示例"、"本领域的强制约束规则"、"顶层设计 vs 关键设计"、"领域设计 vs 关键设计"等指导性文字
- 以 `-->` 结束

**保留**的注释：
- 纯数据性注释

### Frontmatter 更新

每个文件的 frontmatter 都需要更新：
- `last_updated`: 当前日期
- `status`: `"🟡 设计中"`
- 其他字段按各文件具体规则处理

### 规范约束融入

填充任何文件时，必须确保内容与 `.opencode/rules/` 中的规范一致。具体来说：
- 代码结构约束必须反映规范中定义的包名和目录结构
- 命名规范必须采用规范中的约定
- 技术栈选型必须与规范中指定的框架/版本一致（如规范有指定）
- 组件选型必须先对照 `.opencode/rules/component-policy.md`；黑名单组件不得作为新设计选型写入，未列出组件写入 `docs/unlisted-components.md`

## 1. constitution.md 填充规则

### 1.1 标题与引用块

替换 `# {{项目名称}} 全局约束` 中的 `{{项目名称}}`。

保留标题下方的引用块（`> **角色**:` 等），不做修改。

### 1.2 约束条目格式

每条约束一行，格式：
```
- **ID**: 约束内容
```

示例：
```
- **S1**: Spring Boot 3.3.x 单体架构，Maven 多模块管理
- **P1**: RESTful 风格，URI 路径 `/api/v1/{resource}`，HTTP 方法语义化
- **D1**: 主键使用雪花算法 Long 型 ID，全局唯一
```

### 1.3 约束编写原则

- **必须是可执行的硬性规则**，不是建议或最佳实践
- ✅ 正确：`主键使用雪花算法 Long 型 ID`
- ❌ 错误：`建议使用雪花算法作为主键策略`
- 每条约束应当具体到可以直接用于代码审查
- 约束内容从代码中提取事实，不要臆造项目中不存在的规则
- **必须融入 .opencode/rules/ 中的相关规范**

### 1.4 ID 编号规则

- 按分类前缀 + 序号：S1, S2, P1, P2, D1, D2...
- ID 全局唯一（包括后续领域文件中的约束 ID）
- 如果某分类只有 1 条约束，也要编号（如 E1）
- 约束数量按实际提取结果决定，不必凑满模板中的示例数量

### 1.5 分类增删规则

| 项目类型 | 删除分类 | 保留分类 |
|----------|----------|----------|
| 纯后端 | U（UI/UX 约束）、C5-C8 及前端映射表 | S, P, D, E, T, C1-C4 及后端映射表 |
| 纯前端 | D（数据约束）、C1-C4 及后端映射表 | S, P, E, T, U, C5-C8 及前端映射表 |
| 全栈 | 无 | 全部保留 |
| Monorepo | 按子项目类型分别适用上述规则 | 见下方说明 |

**Monorepo 特殊处理**：
- constitution.md 记录所有子项目共享的全局约束，子项目特有的约束用 `{子项目前缀}-` 标注（如 `S1: [admin] Spring Boot 3.3.x 单体架构`）
- architecture.md §3 架构图展示所有子项目及其关系，§3.3 服务清单按子项目分组
- C 分类按子项目分别列出目录结构约束，用子标题区分

同时更新 frontmatter 中的 `constraint_categories` 列表。

### 1.6 D 分类数据库治理规则

D 分类定义项目级数据库硬约束。生成 `constitution.md` 时，必须从
`.opencode/rules/component-policy.md` 和适用的 `.opencode/rules/database-*.md`
中提炼顶层约束，但不要复制数据库专项规则全文。

**必须覆盖的顶层约束维度**：

- 数据库选型必须来自 component-policy 白名单；未列出组件写入 `docs/unlisted-components.md`
- 数据库产品、兼容模式和 SQL 方言必须明确，禁止混用未确认方言
- 数据库接入必须使用统一数据源、连接池和配置中心，禁止业务层裸连接
- 连接串、账号、密码、证书禁止硬编码，生产密钥必须加密管理
- 表必须有主键；业务表和字段必须有业务注释
- 表、字段、索引、约束、序列命名必须遵循适用数据库专项规则
- DDL 必须脚本化并纳入变更流程，禁止应用运行时代码执行 DDL/DCL
- SQL 必须参数化，禁止字符串拼接 SQL 和 MyBatis `${}` 动态拼接

**禁止写入顶层设计的内容**：

- 具体 DDL、字段级清单、完整索引设计
- 某张表的字段类型、长度、默认值、枚举值
- 某条 SQL 或 Mapper XML 的实现细节

这些内容应放入 `docs/modules/{module}/specs/{REQ}/backend-database-design.md`
或模块级 `module-database.md` 当前态地图。

### 1.10 G 分类填充规则

G 分类定义全局基础设施约束，确保公共模块提供的能力被所有业务模块统一使用。

**插入位置**：在 `## 模块间调用约束 (M)` 之后，`## 已澄清问题` 之前。

**约束编写原则**：
- 每条约束必须指明"由哪个公共模块提供"和"业务模块禁止做什么"
- 约束内容应与 architecture.md §9 的清单对应
- 存量项目：从 common/shared 模块扫描提取已有能力，转化为约束
- 新建项目：根据阶段 1.4 收集的全局基础设施决策生成

**示例**：
```markdown
## 全局基础设施约束 (G)

- **G1**: 跨服务共享代码统一放 ins-common 模块，禁止在业务服务中重复实现已有公共能力
- **G2**: 统一异常体系 — BusinessException / ServiceException / SystemException，错误码按模块分段分配（见 architecture.md §9.4）
- **G3**: 统一响应封装 — ApiResponse<T> 由 common 模块提供，Controller 层禁止自行包装
- **G4**: 统一 ID 生成器 — common 模块提供 IdGenerator，禁止各服务自建雪花算法实例
- **G5**: Redis 操作必须通过 common 模块的 RedisClient 封装，内置降级策略（Redis 不可用时降级为本地缓存或直查 DB）
- **G6**: 分布式锁统一使用 common 模块的 DistributedLock 封装，禁止直接操作 Redis SETNX
- **G7**: 请求上下文（traceId / userId / tenantId）通过 RequestContext 统一传播，跨服务调用自动透传
- **G8**: 前端公共层（Axios 拦截器、权限指令、通用 composables）统一放 ins-shared 模块，子应用禁止重复封装
```

**项目类型适配**：
- 纯后端项目：删除 G8
- 纯前端项目：删除 G2-G7 中后端专属项，保留 G1（改为前端公共模块定位）和 G8
- 无 Redis 项目：删除 G5、G6

**同步更新**：frontmatter 的 `constraint_categories` 列表追加 `G`。

### 1.6 领域约束追加规则

当某个领域的约束 ≤ 20 行（不需要创建独立领域文件）时，在 constitution.md 中新增一个领域分类章节。

**插入位置**：在 `## 代码结构约束 (C)` 之前，`## UI/UX 约束 (U)` 之后（如果 U 存在的话，否则在最后一个标准分类之后）。

**格式**：
```markdown
## {领域名称}约束 ({前缀})

- **{前缀}1**: {约束内容}
- **{前缀}2**: {约束内容}
```

**示例**（认证授权领域约束较简单时）：
```markdown
## 认证授权约束 (A)

- **A1**: JWT + Spring Security 认证，Token 有效期 2h，支持 Refresh Token
- **A2**: RBAC 权限模型，角色-菜单-按钮三级控制
- **A3**: 密码使用 BCrypt 加密存储，登录失败 5 次锁定 30 分钟
```

**同步更新**：
- frontmatter 的 `constraint_categories` 列表追加新前缀（如 `[S, P, D, E, T, U, C, A]`）
- 确保新增的约束 ID 与已有 ID 不冲突

**领域前缀参考**（详见 `domain-detection.md`）：

| 领域 | 前缀 |
|------|------|
| 认证授权 | A |
| 消息通信 | M |
| 缓存策略 | CH |
| 文件存储 | ST |
| 定时任务 | SC |
| 日志监控 | O |
| 搜索引擎 | SE |
| 工作流 | W |
| 国际化 | I |
| 微前端 | MF |

### 1.7 C 分类特殊处理

模板中的 C1-C8 是 Spring Boot + Vue 3 的默认结构。如果实际项目使用不同技术栈：

- **完全重写** C 分类的约束内容，反映实际项目的目录结构
- **重写映射表**，反映实际项目的设计文档到代码层的映射关系
- **优先使用 .opencode/rules/ 中定义的代码结构**，如果规范中有明确的目录结构约定
- 如果项目没有明确的分层结构，C 分类可以只保留目录组织规则

### 1.8 已澄清问题

`## 已澄清问题` 章节：如果扫描过程中没有需要记录的问题，删除示例行，保留空表头：
```
| ID | 问题 | 结论 |
|----|------|------|
```

### 1.9 变更记录

更新变更记录表：


新建项目：
```

```

## 2. architecture.md 填充规则

### 2.0 标题与引用块

替换 `# {{项目名称}} 架构蓝图` 中的 `{{项目名称}}`。

保留标题下方的引用块（`> **角色**:` 等），不做修改。

### 2.1 项目定位

`## 1. 项目定位` — 从以下来源提取：
- **存量项目**：README.md 的项目描述、构建文件的 description 字段。如果都没有，从代码结构和业务模块推断
- **新建项目**：需求文件 overview.md 中的系统概述和系统定位

包含系统类型、目标用户、预期规模、系统边界、外部依赖等表格。

### 2.2 技术栈表

`## 2. 技术栈` — 只列项目实际使用的技术：
- 删除项目中不涉及的行（如纯后端项目删除前端行）
- 版本号从构建文件中提取精确版本（存量项目）或从用户选择填充（新建项目）
- 说明列简要描述用途
- **必须与 .opencode/rules/ 中指定的技术栈一致**
- 写入前按 `component-policy.md` 检查：
  - 白名单组件正常写入技术栈，不写入 `unlisted-components.md`
  - 新建项目或设计期计划中的黑名单组件不得写入技术栈，必须在确认阶段要求替换
  - 存量项目已实际使用的黑名单组件如需反映现状，只能标注为规则冲突，不得作为推荐约束，并在生成结果中提示使用 `asdd-rules-review` skill 检查
  - 未列出组件可以写入技术栈，但必须同步追加到 `docs/unlisted-components.md`

### 2.3 系统架构

`## 3. 系统架构` — 包含架构原则、服务拓扑图、服务清单、通信方式等。

**架构图必须使用 ASCII Art，禁止 Mermaid。**

**单体应用示例**：
```
┌─────────────────────────────────────────┐
│              Application                │
├──────────┬──────────┬──────────────────┤
│ Controller│  Service  │   Repository    │
├──────────┴──────────┴──────────────────┤
│              MySQL / Redis              │
└─────────────────────────────────────────┘
```

**微服务示例**：
```
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  Client   │────▶│   Gateway    │────▶│ User Service │
└──────────┘     │  (8080)      │     │ (8081)       │
                 └──────┬───────┘     └──────────────┘
                        │
                        └────────────▶┌──────────────┐
                                      │Order Service │
                                      │ (8082)       │
                                      └──────────────┘
```

要求：
- 展示实际的服务/组件名称和端口
- 用箭头表示调用方向
- 包含中间件（Redis、MQ、ES 等）
- 将模板中的示例内容替换为实际架构图

### 2.4 服务清单

`## 3.3 服务清单`：
- 单体应用：列出主应用 + 依赖的中间件
- 微服务：列出每个服务的名称、职责、端口、依赖
- 端口从配置文件中提取
- 服务清单必须基于仓库拓扑预扫描结果，不得只根据第一个 `src/` 或第一个启动类生成
- Monorepo / 微服务项目必须列出所有后端服务、网关和公共模块；无法确认职责时用扫描证据标注“待确认”，不要丢弃目录
- 微前端项目必须在 §8.2 列出主应用和所有子应用，并关联对应后端服务或网关路由

### 2.5 后续章节

§4-§8 根据项目实际情况选择性填充：
- 数据架构、缓存架构、消息架构、容错架构、前端架构
- 不涉及的章节直接删除
- 保留的章节用实际信息填充

### 2.6 核心架构决策（如存在）

- 只记录结论和理由，不展开方案对比
- 从代码中能推断出的决策（如"选择 MyBatis-Plus 而非 JPA"）
- 通常 3-5 条即可
- 如果无法从代码推断理由，写"项目既有选型"

### 2.9 全局基础设施层（§9）

`## 9. 全局基础设施层` — 定义后端 common 模块和前端 shared 模块提供的全局公共能力。

**本章节始终填充**，不受领域检测结果影响。内容来源：
- 存量项目：从 common/shared 模块扫描提取已有能力
- 新建项目：从阶段 1.4 收集的全局基础设施决策生成

**§9.1 后端 Common 模块**：
- 按包维度列出公共能力清单
- 每行包含：包名、职责、核心类名
- 存量项目：从 common 模块的实际包结构提取
- 新建项目：根据技术栈和架构模式规划
- 纯前端项目：删除本小节

**§9.2 前端 Shared 模块**：
- 按目录维度列出公共能力清单
- 每行包含：目录、职责、核心内容
- 存量项目：从 shared 模块的实际目录结构提取
- 新建项目：根据前端框架和微前端方案规划
- 纯后端项目：删除本小节

**§9.3 降级策略矩阵**：
- 列出所有基础设施组件的降级行为和恢复策略
- 存量项目：从代码中的 fallback/circuit-breaker 配置提取
- 新建项目：根据架构模式和可用性要求规划
- 不涉及的基础设施行删除

**§9.4 错误码分段规则**：
- 按模块/服务分配错误码段
- 通用段（10001-19999）归 common 模块
- 每个业务服务分配一个万位段
- 系统级（90001-99999）归系统异常
- 存量项目：从已有错误码定义提取分段规则
- 新建项目：根据服务清单预分配

## 3. unlisted-components.md 填充规则

### 3.1 创建方式

1. 从 `docs/templates/unlisted-components-template.md` 复制生成 `docs/unlisted-components.md`。
2. 如果文件已存在，保留已有组件条目和确认状态，只补齐缺失表头。
3. 删除 AI 填充指引注释块。
4. 更新 frontmatter 中的 `last_updated` 为当前日期。
5. 追加前按 `ecosystem + group + name` 或包名去重，禁止覆盖已有确认状态。

### 3.2 存量项目初始化

顶层设计生成阶段如能读取到依赖清单和 `.opencode/rules/component-policy.md`：

1. 扫描 `pom.xml`、`build.gradle`、`package.json` 等依赖声明文件。
2. 与 component-policy 的白名单、黑名单比对。
3. 不在白名单且不在黑名单的组件追加到 `docs/unlisted-components.md`，确认状态为“新发现”。
4. 黑名单组件不写入本清单，在生成结果中提示后续使用 `asdd-rules-review` skill 处理。
5. 无法确认用途时，用途说明写“待确认”。

### 3.3 新建项目或设计期计划组件

顶层设计从系统需求、用户选择或基础设施决策中产生计划组件时：

1. 对每个计划组件先查白名单，再查黑名单。
2. 白名单组件不写入 `docs/unlisted-components.md`。
3. 黑名单组件不写入 `docs/unlisted-components.md`，也不得写入 `architecture.md` 技术栈；必须回到用户确认环节选择替代方案。
4. 不在白名单且不在黑名单的组件追加到对应章节，确认状态为“待确认”。
5. 发现来源写“顶层设计生成”，用途说明写该组件在架构中的计划用途。

### 3.4 内容边界

- 本文件只记录未列出组件、包或依赖。
- 不记录合规率、违规详情或合并建议；这些运行时结果写入需求目录的 `rules-review.md`。
- 不覆盖已有条目的确认状态。

## 4. 领域文件填充规则

### 4.1 创建流程

1. 从 `docs/templates/domain-template.md` **复制**到 `docs/domains/` 目录下
2. 重命名为 `{编号}-{domain}.md`（如 `01-auth.md`）
3. 如果 `docs/domains/` 目录不存在，先创建该目录
4. 不要修改 `docs/templates/domain-template.md` 原文件（保留作为模板）

### 4.2 Frontmatter

```yaml
type: top-level-design
document: domain
domain: "{领域中文名}"
status: "🟡 设计中"
related_constraints: ["S1", "E1"]  # 引用 constitution.md 中相关的约束 ID
last_updated: "{当前日期}"
```

`related_constraints` 填充方法：从 constitution.md 中找出与该领域直接相关的约束 ID。

### 4.3 标题和关联约束

```markdown
# {编号} - {领域中文名}

> **关联约束**: S1, E1, ...（引用 constitution.md 中的 ID）
```

### 4.4 概述（§1）

- 1-3 段描述领域目标和范围
- 范围表必须填写"包含"和"不包含"

### 4.5 整体架构（§2）

- ASCII Art 架构图，展示该领域的核心组件和交互
- 核心流程 5-7 步，用编号列表或简单流程图

### 4.6 架构决策（§3）

- 决策表 3-5 行
- 仅当决策涉及复杂方案对比时，取消注释详述子章节并填充
- 简单决策只保留表格

### 4.7 领域约束（§4）

- 使用领域前缀 + 序号（如 A1, A2 表示认证约束）
- 这些约束 ID 必须与 constitution.md 中的 ID 不冲突
- 约束表格式：`| ID | 约束 | 说明 |`

### 4.8 实现指引（§5）

- 列出需要实现该领域功能的模块
- 每个模块一行，说明关注点
- 这是给关键设计阶段的指引，不是具体实现

### 4.9 变更记录

存量项目：
```
| v1.0.0 | {当前日期} | 从代码扫描自动生成 | AI |
```

新建项目：
```
| v1.0.0 | {当前日期} | 基于系统需求初始生成 | AI |
```

## 5. docs/INDEX.md 更新规则

### 5.1 项目名称

替换 `# {{项目名称}} 文档索引` 中的 `{{项目名称}}`。

### 5.2 Frontmatter

```yaml
type: index
scope: global
last_updated: "{当前日期}"
```

### 5.3 AI 指引注释块

删除文件中的 AI 填充指引注释块。

### 5.4 内容更新

- 保持文件已有结构不变
- 更新 `last_updated`
- 删除 AI 指引注释块
- 确保存在“组件治理”章节，并包含 `[unlisted-components.md](./unlisted-components.md)`
- 不修改功能模块表格（那些由其他流程维护）

## 内容边界检查清单

填充完成后，逐项确认以下内容**不存在**于任何顶层设计文件中：

| 禁止内容 | 检查方法 |
|----------|----------|
| 代码片段 | 搜索 ` ```java`、` ```python`、` ```go` 等代码块标记（` ```` 后跟语言名的 ASCII Art 块除外） |
| DDL 语句 | 搜索 `CREATE TABLE`、`ALTER TABLE` |
| API 定义 | 搜索 `@GetMapping`、`@PostMapping`、`GET /api`、`POST /api` 等 |
| 详细时序图 | 搜索 `sequenceDiagram`、`participant` |
| 字段级定义 | 搜索具体的数据库字段列表（如 `| 字段名 | 类型 | 说明 |` 表格） |

这些内容属于关键设计层（`modules/`），不应出现在顶层设计文档中。
