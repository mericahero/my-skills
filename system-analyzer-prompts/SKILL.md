# System Analyzer Prompts Guide

一套完整的提示词工程指南，用于从0开始对企业级Web应用进行深度源码分析并生成完整的系统文档体系。

## 使用场景

当用户需要对以下场景进行系统分析时调用本Skill：
- 从0开始分析一个Web系统的源码结构和业务逻辑
- 对现有系统进行逆向工程，还原架构设计和数据流
- 生成系统级文档（概要设计、详细设计、业务需求、需求详细说明、使用说明）
- 生成模块级深度分析报告
- 建立系统的完整知识图谱和文档索引

## 执行流程

共6个阶段、18组提示词，按顺序执行：

```
阶段0(准备) → 阶段1(探索) → 阶段2(浏览器) → 阶段3(源码分析) → 阶段4(模块文档) → 阶段5(系统文档) → 阶段6(验证)
   P0           P1-P2          P3-P4             P5-P8              P9-P10            P11-P15          P16-P17
```

```
【系统访问信息】
- 系统URL: ____________
- 登录用户名: ____________
- 登录密码: ____________

【源码目录】
- 后端源码根目录: ____________
- 前端源码根目录（如有）: ____________
- 辅助服务源码目录（如有）: ____________

【已知信息】
- 系统名称: ____________
- 技术栈初步判断: ____________
- 已使用的分析工具: ____________

请按以下顺序执行：
1. 探索后端源码根目录的一级和二级结构，识别技术栈（查看package.json/pom.xml/build.gradle等）

```

## 阶段1：项目结构探索

### 提示词 P1：源码目录结构深度扫描

```
基于已确认的项目结构，请对 __[后端源码目录]__ 进行深度扫描：

【任务清单】
1. 列出所有一级目录和文件，标注每个目录的职责推测
2. 识别路由层：扫描 routers/、routes/、controllers/ 等目录下的所有文件
3. 识别模型层：扫描 models/、entities/、domain/ 等目录下的所有文件
4. 识别中间件：扫描 middlewares/、filters/、interceptors/ 等目录
5. 识别配置层：扫描 config/、settings/ 目录
6. 识别定时任务：扫描 jobs/、schedulers/、crons/ 目录
7. 识别工具/脚本：扫描 utils/、tools/、bin/ 目录
8. 读取 package.json/pom.xml/build.gradle 的依赖信息

【输出格式】
按层次输出分析结果，每个文件给出：路径、文件名、预估职责、涉及的业务域
```

### 提示词 P2：前端路由与菜单结构提取

```
基于已确认的前端源码目录 __[前端源码目录]__：

【任务清单】
1. 扫描前端路由配置文件，提取完整的路由表
2. 识别菜单组件，提取菜单层级结构
3. 将路由与菜单进行映射
4. 识别每个路由的权限要求

【输出格式】
一级菜单: XXX
  二级菜单: YYY (/path/to/page) [权限: requireLogin]
    三级菜单: ZZZ (/path/to/subpage)
```


## 阶段3：源码深度分析

### 提示词 P5：核心模块路由文件深度解析

```
基于已识别的路由文件列表，请逐个读取以下类型的源码文件并深度分析：

【目标文件类型】
1. 业务路由文件（routers/*.js 或 controllers/*.java 等）
2. 独立模块路由（如 fundpool/router.js、fofperf/router.js）

【分析维度】
对每个路由文件：
1. 列出所有API端点（HTTP方法 + 路径 + 处理函数名）
2. 分析中间件链（认证、权限检查、参数校验等）
3. 分析每个处理函数的业务逻辑：
   - 接收什么参数
   - 查询哪个数据库/表/集合
   - 执行什么数据操作
   - 返回什么数据
4. 识别核心业务算法
5. 识别外部服务调用

【输出格式】
## 文件: [路径]
### API端点列表
| 方法 | 路径 | 功能 | 中间件 |

### 核心业务逻辑
[详细描述每个端点的处理流程]

### 数据模型引用
[引用的Model/Entity/Table]

### 外部依赖
[调用的其他服务]
```

### 提示词 P6：数据模型Schema深度解析

```
基于已识别的模型文件列表，读取所有数据模型文件：

【分析维度】
1. 对每个模型，提取完整的Schema/字段定义
2. 字段分类：基础信息、时间序列、指标计算、关联关系、权限控制
3. 识别索引设计
4. 识别钩子/触发器
5. 识别虚拟字段/计算属性
6. 分析模型间的关系

【特别关注】
- 时间序列数据的存储结构（如nets[]、returns[]、scale[]数组）
- 嵌套文档结构
- 评分/指标类字段的命名规律
- 权限控制字段

【输出格式】
## 模型: [ModelName]
**数据库**: [库名]
**集合/表**: [集合名]

### 字段清单（按分类）
| 字段名 | 类型 | 必填 | 默认值 | 说明 |

### 索引
| 字段组合 | 类型 |

### 关联关系
[与其他模型的关系]
```

### 提示词 P7：业务算法与工具函数解析

```
基于已识别的工具/算法文件（utils/、bin/、jobs/ 目录），深度分析核心业务算法：

【分析维度】
1. 算法文件的功能说明
2. 输入参数和输出格式
3. 核心计算步骤（逐步拆解）
4. 数学公式
5. 数据流向
6. 异常处理逻辑

【特别关注以下类型的算法】
- 评分/评级算法
- 绩效指标算法（Sharpe、Sortino、MaxDrawdown等）
- 归因分析算法（Brinson、收益分解）
- 优化算法（均值方差、风险平价、Black-Litterman）
- 数据处理算法（频率转换、异常值处理、归一化）

【输出格式】
## 算法: [算法名]
**文件**: [路径]
**功能**: [一句话说明]

### 输入
[参数列表及类型]

### 输出
[返回格式]

### 算法步骤
1. [步骤1]
2. [步骤2]
...

### 核心公式
```
[数学公式]
```

### 伪代码
```
[算法伪代码]
```
```

### 提示词 P8：配置与定时任务分析

```
读取系统的配置文件和定时任务：

【配置文件分析】
1. 系统模块配置：提取功能模块树
2. 功能开关配置：提取各模块启用状态
3. 数据库连接配置
4. 外部服务配置

【定时任务分析】
1. 读取crontab或定时任务配置文件
2. 分析每个任务的：执行时间、执行脚本、功能说明、数据来源、数据目标
3. 绘制定时任务时间表

【输出格式】
## 功能模块配置
| 模块名 | 配置键 | 默认状态 | 说明 |

## 数据库连接
| 连接名 | 数据库类型 | 库名 | 用途 |

## 定时任务
| 时间 | 脚本 | 功能 | 数据来源 | 数据目标 |
```

```
- 在系统中的角色

## 3. 功能清单
| 功能名称 | 前端URL | 后端API | 权限要求 | 说明 |

## 4. 各功能点详细分析
对每个功能点，必须包含以下子节：

### 4.X [功能名称]
#### 功能入口
- 前端URL路径
- 菜单位置
- 对应后端API端点

#### 业务含义

- 解决的痛点
- 业务规则

#### 数据流图
```

```

2. 用户操作步骤2
...

#### 系统详细设计

- **业务逻辑伪代码**
- **数据模型**: 涉及的模型及关键字段

#### 需求说明
- 前置条件
- 输入数据要求
- 输出数据说明
- 业务规则清单
- 异常处理

## 5. 数据模型详细说明
## 6. 接口清单
## 7. 数据流向汇总

【质量要求】
- 每个功能点的详细分析不少于300字
- 数据流图必须清晰展示完整流转路径
- 算法逻辑必须包含公式和伪代码
- 文档总长度不少于800行
```

### 提示词 P10：批量模块文档生成协调

```
系统共有N个模块需要生成深度分析报告。请按以下优先级和依赖关系执行：

【模块优先级】
P0（核心模块，最先处理）:
- 基金数据库（依赖：Fund模型、所有基础数据）
- FOF组合（依赖：基金数据库数据）
- 基金池管理（依赖：基金数据库、FOF组合数据）

P1（业务模块）:
- 研究员考核
- 报表中心
- 资产配置

P2（辅助模块）:
- 项目管理
- 风险监测
- 投资池

P3（基础模块）:
- 用户管理
- 系统管理

【执行策略】
1. 优先并行处理P0模块
2. P0完成后，再并行处理P1和P2
3. 最后处理P3
4. 所有模块完成后，生成系统总览文档

【注意事项】
- 每个模块的源码分析必须在文档生成之前完成
- 模块间共享的数据模型只需分析一次
- 注意识别模块间的调用关系
```

## 使用策略

### 执行顺序

```
阶段0 → 阶段1 → 阶段2 → 阶段3 → 阶段4 → 阶段5 → 阶段6
   P0       P1-P2    P3-P4    P5-P8    P9-P10   P11-P15  P16-P17
```

### 并行策略

- **阶段1**：P1和P2可以并行（后端和前端独立）
- **阶段3**：P5、P6、P7、P8可以并行（路由、模型、算法、配置独立）
- **阶段4**：不同模块的P9可以并行（模块间无依赖）
- **阶段5**：P11-P15可以并行（面向不同读者，内容来源相同）

### 上下文传递

每个阶段的输出应作为下一阶段输入的上下文：
1. P0的输出（项目结构）传递给P1-P2
2. P1-P2的输出（模块清单）传递给P3-P4
3. P3-P4的输出（功能清单）传递给P5-P8
4. P5-P8的输出（源码分析）传递给P9-P10和P11-P15
5. P9-P10的输出传递给P16-P17

### 关键成功因素

1. **先探索后分析**：不要跳过阶段1-2直接读源码
2. **分层阅读源码**：先读路由层建立API地图，再读模型层，最后读算法层
3. **模块间数据共享**：识别跨模块共享的模型，避免重复分析
4. **算法优先**：核心算法是文档最有价值的部分
5. **一致性维护**：使用统一的术语表和命名规范
