# Reviewer 角色定义与检查清单

本文件是 `asdd-code-review` Lead 的维度检查清单来源，定义 8 个 Reviewer 维度的职责、
检查清单和输出要求。

> **Lead-only**：不要把本文件作为所有 code review agents 的共享上下文传入。
> Lead 应先读取本文件，再将每个 reviewer agent 需要的 `dimension_checklist`、
> PR diff、完整文件、关联设计上下文和输出 schema 动态传入。Agent 文件只保留稳定角色纪律；
> 本文件不替代动态 prompt。

---

## R1: code-quality — 代码质量审查

### 职责

审查变更代码的整体质量，包括结构设计、命名规范、设计原则遵循情况和代码复杂度。

### 检查清单

**结构与分层**：
- [ ] 代码是否遵循项目分层架构（Controller → Service → DAO/Mapper）
- [ ] 是否存在跨层调用（如 Controller 直接调用 Mapper）
- [ ] 单个类/文件的职责是否单一（SRP）
- [ ] 方法长度是否合理（建议 ≤ 30 行，超过 50 行必须标记）
- [ ] 类长度是否合理（建议 ≤ 300 行，超过 500 行必须标记）

**命名规范**：
- [ ] 类名是否使用 PascalCase 且为名词/名词短语
- [ ] 方法名是否使用 camelCase 且以动词开头
- [ ] 变量名是否语义清晰，避免单字母（循环变量除外）
- [ ] 常量是否使用 UPPER_SNAKE_CASE
- [ ] 包路径是否符合 `com.picc.{project}.{service}` 规范（后端）
- [ ] 前端文件命名是否符合项目约定（组件 PascalCase，工具 camelCase）

**设计原则**：
- [ ] 是否存在重复代码（DRY 违反）— 3 处以上相似逻辑应提取
- [ ] 是否遵循开闭原则（OCP）— 扩展通过新增而非修改
- [ ] 依赖是否通过注入而非直接实例化（DIP）
- [ ] 是否存在 God Class / God Method
- [ ] 是否存在过深的嵌套（> 3 层 if/for 嵌套）

**复杂度**：
- [ ] 圈复杂度是否合理（单方法建议 ≤ 10，超过 15 必须标记）
- [ ] 认知复杂度是否可接受（逻辑是否容易理解）
- [ ] 是否存在过长的参数列表（> 5 个参数应考虑封装为对象）

## R2: comment-accuracy — 注释准确性审查

### 职责

审查变更代码中注释的准确性、完整性和必要性。确保注释帮助理解代码而非制造混乱。

### 检查清单

**注释与代码一致性**：
- [ ] 方法注释是否准确描述了方法的实际行为
- [ ] 参数说明是否与实际参数名和类型一致
- [ ] 返回值说明是否与实际返回类型和含义一致
- [ ] 异常说明是否列出了方法实际可能抛出的异常
- [ ] 注释中引用的类名/方法名是否仍然存在（未因重构而失效）

**过时注释检测**：
- [ ] 是否存在 TODO/FIXME/HACK 注释指向已完成或不存在的任务
- [ ] 是否存在被注释掉的代码块（应删除而非注释）
- [ ] 是否存在描述旧逻辑的注释（代码已改但注释未更新）
- [ ] 版本号、日期等硬编码注释是否过时

**关键逻辑注释**：
- [ ] 复杂业务规则是否有注释说明"为什么"（不是"做什么"）
- [ ] 非直觉的实现选择是否有注释解释原因
- [ ] 正则表达式是否有注释说明匹配规则
- [ ] 魔法数字是否有注释或已提取为命名常量
- [ ] 临时方案/变通方案是否标注了原因和计划

**注释质量**：
- [ ] 注释是否使用项目约定的语言（中文/英文）
- [ ] 注释是否简洁明了，无冗余废话
- [ ] 公共 API（public 方法/接口）是否有 Javadoc/JSDoc
- [ ] spec-coding 标注是否正确添加（AI 生成的代码）

## R3: test-coverage — 单元测试覆盖与质量审查

### 职责

审查变更代码的单元测试覆盖情况、测试质量和 ASDD TDD 证据。既检查"有没有测试"，也检查"测试好不好"，并在 tasks.md 或 fast-design.md 可用时检查 RED/GREEN 证据是否完整。

### 检查清单

**覆盖率检查**：
- [ ] 新增的 public 方法是否都有对应的测试
- [ ] 新增的 Service 方法是否覆盖了正常流程 + 至少 2 个异常场景
- [ ] 新增的工具函数/Composable 是否有测试
- [ ] 条件分支是否被测试覆盖（if/else、switch 的各分支）
- [ ] 边界条件是否有测试（空值、零值、最大值、集合为空）

**测试质量**：
- [ ] 测试方法名是否描述了被测行为（`shouldXxxWhenYyy` 格式）
- [ ] 每个测试是否只验证一个行为（单一断言原则，相关断言可组合）
- [ ] 断言是否具体（避免只断言 `assertNotNull`，应断言具体值）
- [ ] 测试数据是否有意义（避免 "test", "abc", "123" 等无意义数据）
- [ ] 测试是否独立（不依赖执行顺序，不共享可变状态）

**Mock 合理性**：
- [ ] Mock 的边界是否合理（只 Mock 外部依赖，不 Mock 被测对象内部）
- [ ] Mock 的行为是否与真实实现一致（不过度简化）
- [ ] 是否存在过度 Mock（Mock 了太多东西导致测试无意义）
- [ ] Mock 验证是否必要（避免验证实现细节而非行为）

**测试与 Spec 一致性**：
- [ ] 测试场景是否覆盖了 spec.md 中定义的测试场景表
- [ ] 标准需求测试的业务规则是否与 FRD 一致；fast 需求测试是否与 spec.md / fast-design.md 一致
- [ ] 异常场景测试是否覆盖了设计文档中定义的错误码
- [ ] 如 tasks.md 或 fast-design.md 在审查上下文中，行为变更是否回填 RED 测试、RED 命令、RED 失败原因、GREEN 命令和影响范围验证
- [ ] TDD 证据是否只记录摘要和命令，没有粘贴完整测试日志

**资源管理**：
- [ ] 测试是否正确关闭使用的资源（文件句柄、连接等）
- [ ] 是否使用了 @AfterEach / afterEach 清理测试状态
- [ ] 临时文件/目录是否在测试后清理

## R4: security — 代码安全审查

### 职责

审查变更代码中的安全风险，聚焦 OWASP Top 10 和企业级安全要求。

### 检查清单

**注入防护**：
- [ ] SQL 是否使用参数化查询（禁止字符串拼接 SQL）
- [ ] MyBatis XML 中是否使用 `#{}` 而非 `${}`（除非有充分理由）
- [ ] 用户输入是否经过校验后再使用
- [ ] 是否存在命令注入风险（Runtime.exec、ProcessBuilder 拼接用户输入）
- [ ] 前端是否存在 XSS 风险（v-html 使用未经过滤的用户输入）

**认证与授权**：
- [ ] 新增接口是否配置了权限控制
- [ ] 是否存在越权访问风险（水平越权：访问他人数据；垂直越权：低权限访问高权限接口）
- [ ] Token/Session 处理是否安全
- [ ] 前端路由守卫是否覆盖了新增页面

**敏感数据处理**：
- [ ] 密码是否明文存储或传输
- [ ] 日志中是否打印了敏感信息（密码、身份证号、手机号等）
- [ ] 接口响应是否返回了不必要的敏感字段
- [ ] 前端是否在 localStorage/sessionStorage 中存储了敏感数据
- [ ] 配置文件中是否硬编码了密钥/密码

**数据校验**：
- [ ] Controller 层是否对请求参数做了校验（@Valid、@NotNull 等）
- [ ] 文件上传是否校验了类型和大小
- [ ] 数值参数是否校验了范围
- [ ] 前端表单是否有输入校验（且不仅依赖前端校验）

**其他安全关注**：
- [ ] 是否存在不安全的反序列化
- [ ] 是否存在 SSRF 风险（服务端请求用户提供的 URL）
- [ ] 错误信息是否泄露了系统内部细节（堆栈、SQL、路径）

## R5: api-consistency — 前后端 API 数据结构一致性审查

### 职责

审查前后端 API 接口的数据结构是否对齐，确保前后端契约一致。

### 适用条件

本维度仅在变更同时涉及前端和后端，或变更了 API 接口/DTO/TS 类型时适用。
纯后端或纯前端的内部逻辑变更不触发本维度审查。

### 检查清单

**DTO ↔ TS 类型对齐**：
- [ ] 后端 DTO 字段名与前端 TS interface/type 字段名是否一致
- [ ] 字段类型是否匹配（Java Long ↔ TS number、Java LocalDateTime ↔ TS string 等）
- [ ] 可空性是否一致（Java @Nullable ↔ TS optional `?`）
- [ ] 枚举值是否前后端一致（值和含义都一致）
- [ ] 新增/删除/重命名的字段是否前后端同步更新

**接口契约一致性**：
- [ ] 请求方法（GET/POST/PUT/DELETE）前后端是否一致
- [ ] 请求路径前后端是否一致
- [ ] 请求参数（Query/Path/Body）前后端是否一致
- [ ] 响应结构（包装格式、分页格式）前后端是否一致
- [ ] 错误码前后端是否一致处理

**与 specs/{REQ}/api-design.md 和 module-api.md 一致性**：
- [ ] 实际接口是否与本次需求目录的 api-design.md 中的契约定义一致
- [ ] 是否存在本次需求 api-design.md 中未定义但代码中新增的接口
- [ ] module-api.md 是否同步了当前接口状态、最近变更和代码位置
- [ ] 接口的请求/响应示例是否与实际一致

**数据转换**：
- [ ] 日期格式前后端是否统一（如 ISO 8601）
- [ ] 金额/数值精度前后端是否一致
- [ ] 分页参数命名是否统一（page/pageNum、size/pageSize）
- [ ] 排序参数格式是否统一

## R6: code-simplification — 代码简化审查

### 职责

审查变更代码中是否存在不必要的复杂性，寻找简化机会。

### 检查清单

**过度工程**：
- [ ] 是否存在只有一个实现的接口/抽象类（无扩展计划时不需要）
- [ ] 是否存在不必要的设计模式（如只有一种策略的策略模式）
- [ ] 是否存在过度封装（简单操作被包装了多层）
- [ ] 是否存在不必要的配置化（硬编码即可的值做成了配置）
- [ ] 是否存在预防性编程过度（处理了不可能发生的场景）

**死代码**：
- [ ] 是否存在未被调用的方法/函数
- [ ] 是否存在未被使用的 import/变量/参数
- [ ] 是否存在永远为 true/false 的条件分支
- [ ] 是否存在被注释掉但未删除的代码
- [ ] 是否存在空的 catch 块或空方法体

**可合并逻辑**：
- [ ] 是否存在可以用循环/map 替代的重复代码块
- [ ] 是否存在可以用三元表达式简化的简单 if-else
- [ ] 是否存在可以用 Optional/可选链简化的空值检查链
- [ ] 是否存在可以用 Stream/数组方法替代的手动遍历
- [ ] 是否存在可以合并的相似方法（仅参数不同）

**框架能力利用**：
- [ ] 是否手动实现了框架已提供的功能（如手写分页、手写参数校验）
- [ ] 是否存在可以用注解替代的样板代码（如 @Transactional、@Validated）
- [ ] 前端是否利用了 Vue 3 的响应式能力而非手动管理状态

## R7: type-design — 类型设计审查

### 职责

审查变更代码中的类型设计质量，包括类型安全、接口设计、泛型使用和 DTO 层次。

### 检查清单

**类型安全**：
- [ ] 是否存在不必要的类型转换（强制类型转换、as 断言）
- [ ] 是否存在 any/Object 类型的滥用（应使用具体类型）
- [ ] 泛型是否正确使用（避免原始类型如 `List` 而非 `List<String>`）
- [ ] 枚举是否用于替代魔法字符串/数字
- [ ] 前端 TypeScript 是否开启了 strict 模式相关检查

**接口与抽象设计**：
- [ ] 接口是否遵循接口隔离原则（ISP）— 不强迫实现不需要的方法
- [ ] 抽象类是否合理使用（有共享实现时用抽象类，纯契约用接口）
- [ ] 继承层次是否过深（建议 ≤ 3 层）
- [ ] 是否存在不合理的继承（应该用组合的场景用了继承）

**DTO 层次设计**：
- [ ] 请求 DTO 和响应 DTO 是否分离（不复用同一个类）
- [ ] DTO 是否只包含传输需要的字段（不携带业务逻辑）
- [ ] 是否存在 DTO 和 Entity 混用的情况
- [ ] 分页响应是否使用统一的分页包装类型
- [ ] DTO 之间的继承关系是否合理（避免为了复用字段而强行继承）

**泛型使用**：
- [ ] 泛型参数命名是否有意义（T/E/K/V 或更具描述性的名称）
- [ ] 是否存在可以用泛型消除的重复类型定义
- [ ] 泛型边界（extends/super）是否正确使用
- [ ] 前端是否利用了 TypeScript 的泛型、联合类型、映射类型等高级特性

**Vue 组件类型**：
- [ ] Props 是否使用 TypeScript 类型定义（defineProps 泛型形式）
- [ ] Emits 是否有类型定义
- [ ] Ref/Reactive 是否有明确的类型标注
- [ ] Composable 的返回类型是否明确

## R8: design-consistency — 设计一致性审查

### 职责

审查变更代码是否符合标准需求的 FRD、spec、tasks、详细设计，或 fast 需求的 spec.md、fast-design.md，以及领域设计和顶层设计约束，并检查模块概览是否按当前态同步。
本维度只发现代码与设计/需求之间的偏差，不负责评判代码风格和实现优雅程度。
`docs/requirements/` 只作为可选上游背景和追溯线索，不作为直接判定代码错误的依据。

### 检查清单

**需求与验收一致性**：
- [ ] 标准需求实现是否覆盖 FRD 中的业务规则、主/备选流程、异常流程和验收标准
- [ ] Fast 需求实现是否覆盖 spec.md 和 fast-design.md 中的目标、验收标准、简单设计和不变项声明
- [ ] 实现是否覆盖 spec.md 中定义的本次变更范围
- [ ] 是否遗漏验收标准（AC-*）中的关键行为
- [ ] 是否实现了明确标注为“不做”的范围外功能
- [ ] 如存在 docs/requirements/，是否存在影响追溯的 FRD/spec 上游链路风险（仅提示，不直接判定代码错误）

**任务执行一致性**：
- [ ] 标准需求 tasks.md 中的切片任务是否在代码中有对应实现
- [ ] Fast 需求 fast-design.md 中的内嵌任务是否在代码中有对应实现
- [ ] Fast 需求如涉及浏览器交互，fast-design.md 中声明的关键 `data-testid` 是否在前端代码中落实
- [ ] tasks.md 或 fast-design.md 要求的测试、异常处理、文档同步是否完成
- [ ] 是否存在任务状态标记完成但代码未实现的情况
- [ ] 是否存在代码实现超出任务边界且未在设计中说明的情况

**详细设计一致性**：
- [ ] API 实现是否符合 api-design.md 的接口契约和错误码定义
- [ ] 数据库实现是否符合 backend-database-design.md 的表结构、索引、枚举和迁移设计
- [ ] 后端流程是否符合 backend-detailed-design.md 的 Service 流程、事务、缓存、异常和跨服务调用设计
- [ ] 前端页面/组件/Store 是否符合 frontend-page-design.md 和 frontend-detailed-design.md 的路由、状态、交互和权限设计
- [ ] 前端关键交互是否落实 frontend-page-design.md 的 E2E 可测试性标识契约，且 `data-testid` 未参与业务逻辑、样式或权限判断
- [ ] 是否错误地把 module-backend.md / module-frontend.md 当作详细设计来源，而不是读取 specs/{REQ}/ 下的详细设计
- [ ] Fast 需求是否错误地要求 FRD、完整详细设计或 tasks.md，而不是以 fast-design.md 作为执行文档

**顶层设计与模块边界**：
- [ ] 是否符合 constitution.md 中的全局约束
- [ ] 是否符合 architecture.md 中的服务职责、代码结构、公共能力和依赖关系
- [ ] 是否符合 docs/domains/*.md 中的领域边界、状态流转、业务术语和跨模块协作约束
- [ ] 是否复用 architecture.md §9 中定义的 Common/Shared 公共能力
- [ ] 是否违反模块边界或引入未经设计的新服务/新依赖

**Bug 修复一致性**：
- [ ] Bug 修复是否符合 diagnosis.md 的根因定位和影响范围
- [ ] 修复实现是否覆盖 Bug spec.md 中定义的修复目标、非目标和回归场景
- [ ] 如存在 fix-design.md，修复实现是否遵循最小修复边界、保护约束和回归保护点
- [ ] tasks.md 中要求的回归测试、异常处理和文档同步是否完成

**模块概览同步**：
- [ ] module-api.md 是否同步了新增/变更接口的当前态、代码位置和最近变更
- [ ] module-database.md 是否同步了表、字段、索引和迁移位置
- [ ] module-backend.md / module-frontend.md 是否同步了新增能力、页面、路由、Store 或组件
- [ ] module-*.md 是否只维护当前态索引、变更链和代码位置，没有写入应下沉到 specs/{REQ}/ 的详细流程、完整契约或页面交互细节
