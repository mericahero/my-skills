# asdd-requirements-review

对已拆解的系统需求进行审查澄清和修改调整。

## 定位

本 skill 是 asdd-requirements-decompose 的配套工具，在需求拆解完成后使用。

```
asdd-requirements-decompose（拆解）
    │
    ▼
docs/requirements/（overview.md + req-modules/*.md）
    │
    ▼
asdd-requirements-review（本 skill：审查 + 修改）  ← 可多次调用
    │
    ▼
确认需求无误后 → 使用 asdd-architecture-generate skill → 使用 asdd-frd-generate skill → 使用 asdd-detailed-design-generate skill → ...
```

## 路径约定

- **需求目录**：`<项目根目录>/docs/requirements/`
- **总览文件**：`<项目根目录>/docs/requirements/overview.md`
- **模块需求目录**：`<项目根目录>/docs/requirements/req-modules/`
- **模块需求文件**：`<项目根目录>/docs/requirements/req-modules/{module-name}.md`

## 阶段 0：前置检查与模式判断

### 0.1 前置检查

检查 `<项目根目录>/docs/requirements/overview.md` 是否存在，且 `<项目根目录>/docs/requirements/req-modules/` 下至少有一个 `.md` 文件：

- 不满足以上条件 → 中止，提示用户先使用 asdd-requirements-decompose skill

注意：安装后的空目录可能只包含 `.gitkeep`，不能视为已有拆解产物。

```
⛔ 未找到需求拆解产物。
请先使用 asdd-requirements-decompose skill 将系统需求拆解为模块需求。
```

### 0.2 模式判断

根据用户意图判断工作模式：

| 用户意图 | 模式 |
|---------|------|
| 检查/审查/review/有没有问题/是否一致 | **审查模式**（阶段 1 → 2 → 3） |
| 修改/调整/改/补充/删除/更新某个具体内容 | **修改模式**（阶段 4 → 5） |
| 意图不明确 | 询问用户选择 |

**意图不明确时**，调用 OpenCode `question` tool 询问：

```
请选择操作模式：
  A) 审查模式 — 全面检查需求文件，发现冲突、歧义和不一致
  B) 修改模式 — 对需求进行定向修改
```

### 0.3 加载上下文

无论哪种模式，都需要完整加载现有需求：

1. 读取 `<项目根目录>/docs/requirements/overview.md`
2. 读取 `<项目根目录>/docs/requirements/req-modules/` 下所有 `.md` 文件
3. 构建模块-专题-功能点的完整索引

## 审查模式

### 阶段 1：全面扫描

按以下 7 个维度扫描所有需求文件，每个维度独立检测。

#### 1.1 冲突检测（Conflicts）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 实体归属冲突 | 同一实体被多个模块声明为核心实体 | 🔴 严重 |
| 业务规则矛盾 | 不同模块/专题中对同一业务规则的描述不一致 | 🔴 严重 |
| 循环依赖 | 模块间存在循环调用关系（A→B→C→A） | 🔴 严重 |
| 验收标准冲突 | 不同专题的验收标准互相矛盾 | 🟡 警告 |

#### 1.2 歧义检测（Ambiguities）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 模糊验收标准 | 验收标准缺少具体数值或明确条件（如"快速响应"而非"< 500ms"） | 🟡 警告 |
| 模块边界不清 | 某功能点可合理归属多个模块，但未明确划分 | 🟡 警告 |
| 未定义术语 | 使用了未在任何模块中定义的业务术语 | 🟢 提示 |
| 模糊依赖 | 模块依赖描述笼统（如"依赖用户模块"），未指明具体接口或数据 | 🟡 警告 |

#### 1.3 缺失检测（Missing）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 幽灵模块 | 依赖表中引用的模块在 req-modules/ 中不存在 | 🔴 严重 |
| 无主实体 | 某实体在多个模块中被使用但没有任何模块声明为核心实体 | 🟡 警告 |
| 空专题 | 专题索引中列出但无内容的专题 | 🟡 警告 |
| 缺失验收标准 | 专题没有任何验收标准 | 🟡 警告 |
| 缺失依赖声明 | 模块间明显存在调用关系但未在依赖表中声明 | 🟡 警告 |

#### 1.4 一致性检测（Inconsistencies）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 模块表失同步 | overview.md 模块表与 req-modules/ 目录下的文件不一一对应 | 🔴 严重 |
| 专题索引失同步 | 模块文件内的专题索引表与实际专题内容不匹配 | 🔴 严重 |
| 依赖图不一致 | overview.md 的依赖图与各模块文件中的依赖声明不一致 | 🟡 警告 |
| 优先级不一致 | overview.md 中模块优先级与模块文件内的优先级不一致 | 🟡 警告 |
| 实体列表不一致 | overview.md 核心实体列与模块文件内的实体声明不一致 | 🟡 警告 |

#### 1.5 跨模块检测（Cross-module）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 功能重复 | 不同模块的专题包含实质相同的功能点 | 🟡 警告 |
| 优先级依赖倒挂 | P0/P1 模块依赖 P2 模块 | 🟡 警告 |
| 集成点缺失 | 两个有依赖关系的模块，调用方未描述集成方式 | 🟢 提示 |
| 数据流断裂 | 数据从模块 A 产生，模块 B 消费，但中间没有声明传递路径 | 🟡 警告 |

#### 1.6 完整性检测（Completeness）

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 最低覆盖缺失 | 模块文件未满足最低覆盖检查清单（模块目标/专题索引/核心实体/依赖/优先级/需求来源） | 🔴 严重 |
| overview 必须项缺失 | overview.md 缺少必须覆盖项（系统定位/模块表/拓扑图/依赖图/系统边界/拆解历史） | 🟡 警告 |

#### 1.7 微服务拆分合理性检测（Microservice Decomposition）

基于 Java/Spring Cloud 微服务架构最佳实践和 DDD 方法论，检测模块拆分是否遵循微服务拆分原则：

| 检测项 | 说明 | 严重度 |
|--------|------|--------|
| 限界上下文重叠 | 两个或多个服务操作同一聚合根，或共享同一限界上下文，违反 DDD 限界上下文独立性原则 | 🔴 严重 |
| 服务粒度失衡 | 过细：模块仅 1-2 个 Topic 且无独立数据实体（建议合并）；过粗：模块超过 8 个 Topic 或跨越 3 个以上子域（建议拆分） | 🟡 警告 |
| 同步调用链过长 | 同步调用链（Feign）超过 3 层（A→B→C→D），形成分布式单体风险 | 🔴 严重 |
| 共享数据模型未拆分 | 多个服务共享同一数据实体或数据模型，违反数据库独占原则（constitution.md S2） | 🔴 严重 |
| 通信模式未声明 | 模块间依赖关系未标注通信模式（sync-Feign / async-MQ / event-driven），无法评估耦合程度 | 🟡 警告 |
| 缺少限界上下文标注 | 模块未标注对应的 DDD 限界上下文名称，无法验证领域边界对齐 | 🟡 警告 |
| 基础设施能力混入业务服务 | 通用基础设施能力（认证鉴权、网关路由、日志审计、消息中间件封装）混合在业务服务中，而非独立为基础设施服务 | 🟡 警告 |

### 阶段 2：问题报告与用户澄清

#### 2.1 问题汇总

将检测到的问题按严重度分批展示。

**第一批：严重问题（🔴）** — 必须逐项确认：

```
🔴 发现 {N} 个严重问题，需要逐项确认：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[C1] 实体归属冲突
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
实体 `Payment` 同时被以下模块声明为核心实体：
  - order-flow（T3 订单支付）
  - billing（T1 账单管理）

位置：
  - req-modules/order-flow.md 核心实体表
  - req-modules/billing.md 核心实体表
```

对每个严重问题，调用 OpenCode `question` tool 提供修复选项：

```
如何处理实体 Payment 的归属冲突？
  A) 归属 order-flow — billing 模块通过接口调用获取支付信息
  B) 归属 billing — order-flow 模块通过接口调用发起支付
  C) 拆分 — 创建独立的 payment 模块管理支付实体
  D) 跳过 — 暂不处理，后续再决定
```

**第二批：警告问题（🟡）** — 支持批量确认：

```
🟡 发现 {N} 个警告问题：

[W1] 模糊验收标准
  - user-auth T1 AC-003: "登录响应时间应合理" → 建议改为 "< 500ms"
  位置：req-modules/user-auth.md T1 验收标准

[W2] 优先级依赖倒挂
  - order-flow (P0) 依赖 notification (P2)
  位置：req-modules/order-flow.md 模块依赖

...

请逐项确认，或输入"全部接受建议"批量处理。
```

**第三批：提示信息（🟢）** — 仅展示，不要求确认：

```
🟢 {N} 条提示信息（仅供参考，无需确认）：
  [I1] 术语 "风控评分" 仅在 risk-engine 中出现，建议在 overview.md 术语表中定义
  [I2] order-flow 和 billing 的集成方式未明确，建议后续设计时关注
```

#### 2.2 无问题时的处理

如果扫描未发现任何问题：

```
✅ 需求审查完成，未发现问题。

扫描范围：
  - overview.md
  - {N} 个模块需求文件
  - {M} 个专题

检测维度：冲突 / 歧义 / 缺失 / 一致性 / 跨模块 / 完整性 / 微服务拆分合理性
结果：全部通过 ✓

需求文件可以进入下一步流程（使用 asdd-architecture-generate skill → 使用 asdd-frd-generate skill → 使用 asdd-detailed-design-generate skill）。
```

### 阶段 3：自动修复

#### 3.1 收集用户决策

将阶段 2 中用户的所有决策汇总：

```
📋 修复计划：

| 编号 | 问题 | 用户决策 | 影响文件 |
|------|------|---------|---------|
| C1 | Payment 实体归属 | 归属 order-flow | order-flow.md, billing.md |
| W1 | 模糊验收标准 | 改为 < 500ms | user-auth.md |
| W2 | 优先级倒挂 | 接受建议 | overview.md |

确认执行以上修复？
```

#### 3.2 批量修复

用户确认后，按以下顺序修复：

1. **变更深度评估** — 对所有待修复项进行整体评估，判断本轮修复的变更深度（参见关键原则「修改深度评估」）。如果任一修复项触发结构性重写条件，则所有受影响的章节都按结构性重写处理
2. **各模块文件** — 修改 req-modules/*.md 中的具体内容；结构性重写时，对受影响模块的依赖声明、专题内容整节重写
3. **overview.md** — 同步更新模块表、依赖图等汇总信息；结构性重写时，依赖图和拓扑图必须基于最新状态完整重新绘制
4. **拆解历史** — 在 overview.md 拆解历史表追加一行，记录本次审查的修改

#### 3.3 修复验证

修复完成后，对修改的文件重新执行相关检测项，确认问题已解决。

```
✅ 审查修复完成

已修复：{N} 项问题
跳过：{M} 项（用户选择暂不处理）

已修改文件：
  - <项目根目录>/docs/requirements/req-modules/order-flow.md
  - <项目根目录>/docs/requirements/req-modules/billing.md
  - <项目根目录>/docs/requirements/req-modules/user-auth.md
  - <项目根目录>/docs/requirements/overview.md

验证结果：修复后的文件通过所有相关检测 ✓
```

## 修改模式

### 阶段 4：理解修改意图

#### 4.1 定位修改目标

根据用户输入定位要修改的内容：

| 用户说 | 定位 |
|--------|------|
| "修改 user-auth 模块" | → `req-modules/user-auth.md` 整个文件 |
| "修改 user-auth 的 T2" | → `req-modules/user-auth.md` 的 T2 专题 |
| "修改 user-auth 的 T2 的 F2.3" | → `req-modules/user-auth.md` T2 专题的 F2.3 功能点 |
| "调整模块依赖关系" | → overview.md 依赖图 + 相关模块的依赖声明 |
| "修改系统定位" | → overview.md 系统定位部分 |

**如果用户未明确指定目标**，列出可修改的范围供选择：

```
请选择要修改的范围：
  A) 系统总览（overview.md — 系统定位/模块划分/架构/依赖）
  B) 具体模块（选择模块后继续选择专题）
```

如果用户选择具体模块，进一步列出该模块的专题：

```
user-auth 模块包含以下专题：

| 专题 | 名称 | 功能数 | 状态 |
|------|------|--------|------|
| T1 | 用户登录 | 4 | 🔴 |
| T2 | Token 管理 | 3 | 🔴 |
| T3 | 权限管理 | 5 | 🔴 |

请选择要修改的专题（输入编号），或输入"全部"修改整个模块：
```

#### 4.2 理解修改内容

用户的修改请求分类：

| 修改类型 | 影响范围 |
|---------|---------|
| **内容修改** — 改功能描述、业务规则、验收标准 | 目标模块文件 |
| **新增专题** — 在已有模块中增加新专题 | 目标模块文件 + overview.md 专题数 |
| **删除专题** — 删除已有专题 | 目标模块文件 + overview.md 专题数 |
| **移动功能** — 将功能点从一个模块移到另一个 | 源模块文件 + 目标模块文件 + overview.md |
| **调整依赖** — 新增/删除/修改模块间依赖 | 相关模块文件 + overview.md 依赖图 |
| **调整优先级** — 修改模块或专题的优先级 | 目标模块文件 + overview.md 模块表 |
| **新增模块** — 增加新的模块 | 新建模块文件 + overview.md 模块表 |
| **删除模块** — 删除已有模块 | 删除模块文件 + overview.md 模块表 + 相关模块的依赖引用 |
| **调整服务边界** — 基于 DDD 限界上下文重新划定模块边界，迁移功能点和实体归属 | 相关模块文件 + overview.md（触发结构性重写） |
| **合并过细服务** — 将粒度过小（< 2 Topic 且无独立实体）的服务合并到关联服务中 | 被合并模块文件删除 + 目标模块文件更新 + overview.md（触发结构性重写） |
| **拆分过粗服务** — 将职责过多（> 8 Topic 或跨 3+ 子域）的服务拆分为多个独立服务 | 原模块文件删除 + 创建多个新模块文件 + overview.md（触发结构性重写） |

### 阶段 5：执行修改

#### 5.1 展示修改计划

在执行前，先评估变更深度（参见关键原则「修改深度评估」），然后展示修改计划供用户确认：

```
📋 修改计划：

**修改类型**：{类型}
**目标**：{模块/专题/功能点}
**变更深度**：{局部微调 / 结构性重写}

**具体变更**：
  1. req-modules/user-auth.md T2:
     - 删除 F2.3 Token 黑名单（用户指示移除）
     - 新增 F2.3 Token 撤销（用户新增需求）
     - 更新验收标准 AC-005

  2. overview.md:
     - 更新 user-auth 模块的核心功能数（3 → 3，替换不变）

{如果是结构性重写，额外展示}：
⚠️ 本次修改触发结构性重写，以下章节将被完整重写：
  - overview.md: 依赖关系图、微服务拓扑图
  - req-modules/order-flow.md: 模块依赖声明
  - ...

确认执行？
```

#### 5.2 执行修改

用户确认后：

1. **变更深度评估** — 根据修改类型和影响范围，判断变更深度（参见关键原则「修改深度评估」）。对于调整依赖、移动功能、新增/删除模块等操作，通常属于结构性重写
2. **修改目标文件内容** — 局部微调直接修改目标文本；结构性重写时，对受影响模块文件的相关章节（依赖声明、专题内容等）进行整节重写
3. **同步更新 overview.md** — 结构性重写时，模块表整表重新生成，依赖图和拓扑图基于最新状态完整重新绘制，而非在原内容上追加或删改
4. 确保专题编号和功能点编号连续（如果是删除操作，后续编号不重排，保持稳定引用）
5. 在 overview.md 拆解历史表追加一行，记录本次修改

#### 5.3 一致性校验

修改完成后，自动对受影响的文件执行一致性检测（阶段 1.4），确保修改未引入新的不一致：

```
✅ 修改完成

已修改文件：
  - <项目根目录>/docs/requirements/req-modules/user-auth.md
  - <项目根目录>/docs/requirements/overview.md

一致性校验：通过 ✓
```

## 关键原则

### 修改深度评估

**每次修改前必须评估变更深度，决定是局部微调还是结构性重写。**

#### 变更深度分类

| 深度 | 判断标准 | 处理方式 |
|------|---------|---------|
| **局部微调** | 仅影响单个功能点的描述、验收标准数值、拼写错误等，不改变模块间关系 | 直接修改目标文本 |
| **结构性重写** | 影响到模块边界、依赖关系、微服务拓扑、实体归属、模块职责定位中的任意一项 | 对受影响的**整个章节**进行深度重写，而非在原文上打补丁 |

#### 结构性重写的触发条件（满足任一即触发）

- 模块间依赖关系发生变化（新增、删除、方向改变）
- 实体归属发生迁移（从模块 A 移到模块 B）
- 模块职责描述需要调整（不再是原来的定位）
- 微服务拓扑受到影响（服务合并、拆分、通信方式变化）
- 功能点跨模块迁移
- 优先级变化导致依赖链路需重新评估

#### 结构性重写的执行要求

当判定为结构性重写时：

1. **overview.md 的依赖关系图**：不是在原图上改一条线，而是基于当前所有模块的最新依赖关系**重新绘制完整的依赖图**
2. **overview.md 的微服务拓扑图**：不是修改某个节点的标注，而是基于当前所有模块的最新归属**重新绘制完整的拓扑图**
3. **overview.md 的模块总表**：不是改一个单元格，而是基于当前状态**重新生成完整表格**，确保所有列的数据一致
4. **模块文件的依赖声明**：所有受影响模块的上下游依赖**整节重写**，而非追加一行
5. **模块文件的专题内容**：如果专题的业务逻辑因依赖变化而受影响，**重写专题描述和验收标准**，使其在新的模块关系下仍然合理

**核心要求：结构性重写后的内容质量不得低于 asdd-requirements-decompose 初次生成的质量。** 如果修改后某个章节读起来像是打了补丁的拼凑文本，说明重写不够彻底。

### 非破坏性修改

- 删除功能点时，编号不重排（F2.1, F2.2, F2.4 — 保留空位），确保外部引用稳定
- 删除专题时同理（T1, T2, T4 — 保留空位）
- 已经进入后续流程（FRD 已生成）的专题，修改时需提示用户同步更新 FRD

- `#N` — asdd-requirements-decompose 的输入轮次
- `RN` — 审查修复轮次
- `MN` — 手动修改轮次

### 与 asdd-requirements-decompose 的边界

| 操作 | 使用哪个 skill |

| 修改具体的需求内容 | **asdd-requirements-review**（修改模式） |
| 修正审查发现的问题 | **asdd-requirements-review**（审查模式自动修复） |
